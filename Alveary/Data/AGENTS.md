## Data Models

These instructions cover the SwiftData models under `Alveary/Data/`.

- `AgentThread.name` stores the visible thread label, while `AgentThread.hasCustomName` distinguishes a manual rename from the default untitled state. Manual thread rename flows must set `hasCustomName`, and thread auto-naming should only fire while the thread is still effectively untitled (`!hasCustomName && trimmedName == "New thread"`). Conversation auto-titling is a separate rule: the first user message may set `Conversation.title` whenever `customTitle == nil`, even if the thread already has a non-default name. Thread rename cascades to the main conversation's `title` when it still has its default name (`customTitle == nil`); do not add a separate rename affordance for the sole conversation when only one exists.

## Persistence Invariants

These are persistence contracts backed by SwiftData fields. Treat them as hard constraints unless the work explicitly includes a coordinated migration.

- Archived-thread restore uses persisted per-conversation `pendingRestoreContext`, not provider resume. Restoring a thread should regenerate that summary from saved `ConversationEventRecord`s, hydrate it back into `ConversationState.stagedContext` when the conversation view model is recreated, send it only through the existing staged-context path on the next outbound message, and clear the persisted field when the user dismisses it or that send succeeds.
- `Project.remoteName` and `Project.gitRemote` are a paired invariant. Persist and update them together, and have Git/worktree/GitHub flows use the stored `remoteName` instead of rediscovering a remote ad hoc.
- `AgentThread.model` is the per-thread model override. It mirrors the `permissionMode`/`effort` pattern for thread-scoped picker state, but with a different nil semantic:
    - **Store `nil` for "use the provider's default model".** The composer picker's `"default"` value is a UI sentinel that `applyModelChange` translates to `nil` before persisting — never write the literal string `"default"` into `AgentThread.model`, or the adapter will pass `--model=default` to the CLI.
    - **Seed from `AppSettings.defaultModel` at thread creation.** `SidebarViewModel.createThread` reads the app-wide default and maps `"default"`/unknown values back to `nil` on the new `AgentThread`, matching how it seeds `effort` and `permissionMode`.
    - **Read the live DB field in bindings, not a transient state mirror.** `ChatView.selectedModelBinding` must read `conversation.thread?.model` so the picker reflects the persisted value across view-model re-inits and session forks; do not reintroduce a parallel `ConversationState.selectedModel` cache.
- `AgentThread.effort` is model-scoped, not universal. Both the set of acceptable values *and* the preferred default depend on which model the thread is using.
    - **Configure per-model levels in `AppSettings.effortLevelsByModel`.** Any model not listed there falls back to `AppSettings.defaultModelEffortLevels`. Extend this map (and `supportedEffortLevels` if adding a brand-new level) rather than introducing a separate carve-out flag per model. **Why:** the current case is Opus 4.7's `xhigh`, but the same shape will cover future divergences without another bespoke rule.
    - **Configure per-model preferred defaults in `AppSettings.defaultEffortLevelsByModel`.** Unlisted models fall back to the universal `AppSettings.defaultEffortLevel`. Seed/coerce sites read through `AppSettings.defaultEffortLevel(forModel:)` so a fresh install on Opus lands on `xhigh` instead of dragging `medium` from the untouched Settings field. **Why:** "top-tier level the model is designed for" is a per-model property, not a global one.
    - **Coerce in lockstep with model changes.** `ConversationViewModel.applyModelChange` and `SettingsViewModel.defaultModel`'s setter must reset `effort` to `AppSettings.defaultEffortLevel(forModel: newModel)` whenever the stored value is not in `supportedEffortLevels(forModel: newModel)`; the Settings setter additionally upgrades an untouched `AppSettings.defaultEffortLevel` to the new model's preferred default so the Settings picker reflects the same value a fresh thread will receive. Both writes have to happen under the same save so SwiftUI sees model + effort invalidate on one render tick and only **one** `reconfigureSession()` fork fires. **Why:** passing a level the CLI rejects (`--effort xhigh --model sonnet`) fails the next spawn; doing the coercion in a second save would also churn the session twice for a single user gesture.
    - **Seed new threads through `SidebarViewModel.seedEffortLevel(forModel:)`.** The helper returns the user's Settings effort when they customized it to something valid for the thread's model, otherwise the per-model preferred default. Do not inline `settingsService.current.effort` at the thread-seed call site — it skips the per-model upgrade and regresses the fresh-install-on-Opus case back to `medium`.
    - **Filter the composer dropdown at render time.** `ChatComposerPanel.visibleEffortLevels` intersects the provider's full effort list with `supportedEffortLevels(forModel: selectedModel)`. Do not replace the provider list or the intersect — a future provider that ships a narrower set should remain authoritative on its own ceiling.

## Model Context Helpers

- `ModelContext+Resolve.swift` hosts shared typed lookup helpers such as `resolveThread(id:)`. Call sites should prefer `modelContext.resolveThread(id:)` over ad-hoc `modelContext.model(for: id) as? AgentThread` casts so the cast lives in one place. Add sibling resolvers here when another model grows a second call site.
