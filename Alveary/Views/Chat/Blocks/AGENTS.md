## Chat Block Rendering

These instructions cover transcript block view code under `Alveary/Views/Chat/Blocks/` — `ChatBlocks.swift` and its `ChatBlocks+*.swift` companions. Broader chat-specific conventions live in sibling docs: bubble and composer rendering / interaction contracts in `Alveary/Views/Chat/AGENTS.md`; transcript scrolling and follow-mode in `Alveary/Views/Chat/Transcript/AGENTS.md`.

## Tool Transcript Blocks

- Tool transcript blocks come in two shapes, keyed off the `ChatItem` case:
    - **`ToolGroupBlock`** renders `.toolGroup(tools:)`. Two or more tools show a disclosure header with a pluralized, sentence-case label like `Reading 3 files, searching for 2 patterns`; expanding lists each tool as an `InlineToolRow` (individually re-expandable for input/output details). Phrasing is category-driven in `ToolGroupBlock.toolCategorySummary(for:count:)` — `Read` → "Reading N file(s)", `Grep`/`Glob` fold into `Search` → "Searching for N pattern(s)", `ToolSearch` → "Searching for N tool(s)", `WebFetch` → "Fetching from the web", `WebSearch` → "Searching the web", MCP calls fold into "N MCP call(s)". Only the first category is capitalized; subsequent categories are lowercased via `ToolGroupBlock.lowercasedFirstLetter` so the joined list reads like a sentence. A group of size 1 renders the single tool directly inside the pill with no group header — the "group" framing is only visually meaningful when folding multiple entries, and this matches the Claude CLI behavior.
    - **`StandaloneToolRow`** renders `.standaloneTool(tool:)` — one pill per tool call for mutating/shell work (Write/Edit/MultiEdit/NotebookEdit/Bash and any unknown tool). Never re-wrap these in a parent `ToolGroupBlock`.
- Centered transcript notes are a separate, reusable surface:
    - **Use `CenteredTranscriptNote` for subtle lifecycle text.** `Interrupted`, successful plan-mode transitions, and denied `ExitPlanMode` render as full-width centered caption text, not as bubbles or tool pills.
    - **Keep the primitive generic.** `TurnInterruptedNote` is only a thin wrapper around the reusable centered-note view; new subtle lifecycle rows should use `CenteredTranscriptNote` directly instead of cloning the styling.
- `ToolApprovalBlock` renders `.toolApproval(approval:)` as an assistant-side approval surface:
    - **Keep content concise.** Show the approval title, display name, one-line summary, and Approve/Deny actions; do not dump full tool input JSON.
    - **Read tool-specific wording from the request.** Approval title, optional display-name visibility, and approve/deny button labels must come from `ToolApprovalRequest.approvalPromptCopy`, so tools like `ExitPlanMode` can ask "Ready to leave plan mode?" and hide the redundant middle line without view-local `toolName` switches.
    - **Keep only the live approval interactive.** When a newer deferred tool use replaces an older unresolved row, render the older row as non-interactive past-tense `Superseded` instead of showing active approval buttons on multiple rows at once.
    - **Block approvals behind unanswered prompts.** If the same conversation still has an unanswered `AskUserQuestion`, disable the approval controls rather than letting the transcript advance through the deferred-tool resume path first.
    - **Offer session approval only for supported scopes.** Bash can expose a split session-approval control with `Approve exactly` and `Approve group`, but only when the command has a clear subcommand-like group and is not a compound shell command; file-edit approvals stay exact-path only and render a single `Approve for session` action. Unsupported tools show only the one-shot Approve/Deny actions.
    - **Keep the split control self-contained.** The session-approval menu lives inside the button chrome with a divider and trailing caret, so changing the selected scope does not add extra transcript rows or explanatory text.
    - **Collapse after resolution starts.** `.approving` / `.approved` hides Deny and shows disabled past-tense `Approved`; `.denying` / `.denied` hides Approve and shows disabled past-tense `Denied`; session-approved states hide the other actions and show only the resolved past-tense session button (`Approved for session`, `Approved exactly`, or `Approved group`). Nil/non-pending status should keep actions disabled so the same approval cannot be submitted twice from the UI.
- `PromptBlock` should hug its content instead of filling the whole transcript width:
    - **Measure the widest question card.** Use that width as the shared width for every question card in the prompt so stacked plan-mode questions line up cleanly.
    - **Measure the real card before width sync.** Attach width readers to the visible question cards before any shared-width frame is applied; measuring a hidden surrogate or an already-expanded frame can leave the cards mismatched and the bubble full width.
    - **Keep the outer bubble non-greedy.** Do not use trailing spacers or other full-width footer layout that forces the prompt bubble to expand to `transcriptBubbleMaxWidth`; the bubble should size to the widest question card, capped by the bubble max width.
    - **Center choice glyphs against the whole option stack.** Radio buttons and checkboxes should align to the vertical center of the title+description pair, not the first text line.
    - **Include a custom-response path.** `AskUserQuestion` cards should synthesize an `Other` choice when the question allows custom responses, reveal an inline text field in the option's content area when selected, and serialize the typed text instead of the literal word `Other`.
    - **Focus the inline custom-response field immediately.** When `Other` becomes selected, move focus into the inline text field in the same interaction so typing can start without an extra click.
    - **Snapshot the selected custom-response state.** Keep a dedicated prompt snapshot with `Other` selected so regressions in the inline field layout are visible.
    - **Render answered prompts as structured Q/A rows.** The submitted state title is `Submitted responses`, with each question and answer on separate lines and 8pt between Q/A pairs; do not collapse them back into one paragraph-like text blob.
- Bubble widths are capped via the `transcriptBubbleMaxWidth` `EnvironmentKey` in `ChatBlocks.swift`, not a hard-coded 720pt. `ChatTranscriptView` measures its scroll-container content width and runs it through `adaptiveTranscriptBubbleMaxWidth(for:)` before publishing `.environment(\.transcriptBubbleMaxWidth, ...)`; every transcript bubble (`ToolGroupBlock`, `StandaloneToolRow`, `SubAgentBlock`, `TaskListBlock`, `PromptBlock`, `AssistantBubble`, `StreamingBubble`, `ActiveTurnThinkingIndicator`, `ErrorBanner`) reads it via `@Environment` and applies it to its outer `.frame(maxWidth: bubbleMaxWidth, alignment: .leading)`. The adaptive rule is: roomy transcripts cap inbound bubbles around two-thirds width, mid-sized transcripts stop shrinking below a comfortable ~640pt reading width, and compact transcripts fall back to the old near-edge behavior by leaving a 24pt trailing gutter. Fallback is `.infinity` so the first paint before the geometry callback lands isn't clamped to zero. Do not replace this with a fixed constant. `UserBubble` is the one exception: it keeps its own `.frame(maxWidth: 640, alignment: .trailing)` because outbound bubbles are narrower and right-aligned by design.
- Hug-to-content is a load-bearing layout rule for bubbles. `ToolHeaderRow`'s HStack, `ToolGroupBlock`'s multi-entry header HStack, and `DetailCodeBlock`'s `ScrollView` intentionally do **not** contain a trailing `Spacer` or a `.frame(maxWidth: .infinity)`; adding either makes the HStack/ScrollView greedy and defeats the hug-width behavior, which then forces the enclosing bubble to grow to `bubbleMaxWidth` even for short summaries or tiny Input/Output snippets. The bubble's own outer `.frame(maxWidth: bubbleMaxWidth)` still caps it on the long side, and `HighlightedCodeBlock` / `DetailCodeBlock` scroll their content horizontally inside the bubble when it would overflow. Apply the same rule to any new tool-transcript chrome.
- Single-tool bubbles (`ToolGroupBlock` single-entry, `StandaloneToolRow`, `InlineToolRow`) toggle via a `Button` whose label wraps `ToolHeaderRow`, with `ToolDetails` rendered as a sibling below. `InlineToolRow` is the single canonical inline variant used by both `ToolGroupBlock`'s expanded list and `SubAgentBlock`'s nested tool list — do not reintroduce a second `SubAgentToolRow` type since the two surfaces share the same header-button + sibling-details layout. Do not revert to `.highPriorityGesture(TapGesture())` on the whole bubble: when the expanded `DetailCodeBlock` (horizontal `ScrollView` + `.textSelection(.enabled)`) is mounted, a bubble-wide tap gesture contended with AppKit's text-selection machinery and could freeze expand/collapse taps. The header-button pattern mirrors the multi-entry `ToolGroupBlock` branch and lets expanded content own its own interactions (text selection, horizontal scroll) without contention.
    - **For the bubble-chrome variants (`ToolGroupBlock` single-entry and `StandaloneToolRow`), pull the bubble padding *inside* the Button's label** — `.padding(.horizontal, chatBlockPadding).padding(.vertical, chatVerticalPadding)` — instead of padding the outer VStack. Short tool summaries like `ToolSearch` "Searching for tool `X`" leave the header HStack only ~200pt wide; if the padding stays outside, clicks on the bubble's chrome-side padding miss the Button and the row feels hard to expand/collapse. With padding inside the label, the Button's hit area covers the entire header band of the bubble. `ToolDetails` then re-adds `.padding(.leading, toolDetailLeadingInset + chatBlockPadding)` / `.padding(.trailing, chatBlockPadding)` / `.padding(.bottom, chatVerticalPadding)` to keep the details aligned under the icons and match the bubble's vertical rhythm, since the outer VStack no longer pads them.
- Tool bubble vertical padding (`chatVerticalPadding`) is shared with plain user/assistant text bubbles in `ChatSupplementaryViews.swift` so a collapsed tool row hugs the same height as its neighboring text bubble. Horizontal padding differs (`chatBlockPadding = 14` for tool chrome, `chatBubbleHorizontalPadding = 12` for text bubbles) — that split is intentional. Keep `chatVerticalPadding` as the single source of truth for vertical rhythm.
- Bubble chrome (rounded fill + width cap) is applied via `.bubbleBackground(maxWidth:)` — shared `View` extension in `ChatBlocks.swift`. Expand/collapse animation has its own set of invariants documented in the **Expand/Collapse Animation** sub-section below.
- Output-pager behavior is centralized in `ToolOutputView`. `Bash` (10 lines) and `Read` (20 lines) tail their output and expose a `Show N more` button that extends the window upward. Other tools render the full output via `DetailCodeBlock` with no paging. Keep the tail-not-head rule for Bash so streaming output reveals the latest line at the bottom.
- Thinking stream events are intentionally dropped by the grouper; the `ChatBlocks+*.swift` family therefore does not provide a `ThinkingRow`/`ThinkingBlock`. The live `ActiveTurnThinkingIndicator` in `ChatSupplementaryViews.swift` remains the only "thinking" affordance and covers the empty-turn case.
- The chevron/status/summary column across every tool bubble shape (`StandaloneToolRow`, `ToolGroupBlock` single-entry, `InlineToolRow`, *and* the multi-entry `ToolGroupBlock` header) is driven by two shared pieces in `ChatBlocks+ToolHeaderRow.swift`:
    - **`ToolStatusIndicator(isError:isComplete:)`** is the single source of the icon/spinner. Error → red `xmark.circle.fill`; complete → green `checkmark.circle.fill`; otherwise → `ProgressView()` at `.controlSize(.small).scaleEffect(0.75)`. Pin **both** width and height on the outer `Group` via `.frame(width: 18, height: 16, alignment: .center)`. The height is load-bearing: `ProgressView().controlSize(.small)` reserves ~16pt of intrinsic layout height (scaleEffect is only a render transform, it doesn't shrink the layout box), while the SF Symbol at the ambient body font is intrinsically ~13pt tall. Without a fixed height, the HStack grew during the in-progress state and then shrank on spinner→checkmark, which caused a visible vertical nudge in the enclosing bubble whenever a tool group rapidly toggled between working and success as additional tool calls streamed in. 16pt matches the spinner's natural layout size, so pinning there keeps rows at the height they already had while the spinner was present instead of expanding further — icons render centered within that slot at their intrinsic ~13pt size.
    - **`ProgressView()` over a custom SwiftUI spinner.** A prior iteration used a SwiftUI `Circle().trim()` + `withAnimation(.repeatForever)` in `.onAppear` to avoid `NSProgressIndicator`'s first-frame warmup and to keep snapshot rotation deterministic. That caused thread-open renders to land with a blank transcript until the user scrolled — a never-ending SwiftUI animation on a LazyVStack row interacts badly with `scrollPosition` / `defaultScrollAnchor` layout coordination. Stay on `ProgressView`; its rotation lives in the AppKit layer and doesn't disturb SwiftUI layout.
- Multi-entry `ToolGroupBlock` header mirrors the single-tool header visually (same chevron, same `.subheadline.weight(.semibold)` + `.primary` text treatment), but its aggregate icon goes through `DebouncedToolStatusIndicator` rather than the raw `ToolStatusIndicator`. Its aggregate state over `tools` uses error-first priority: `aggregateIsError = tools.contains(where: \.isError)`, `aggregateIsComplete = tools.allSatisfy(\.isComplete)`. Error wins over in-progress so a partial failure reads as a failure even while siblings are still running; the trailing `N failed` tag (the only red-tinted element in the header row) remains as secondary detail. The debouncer only holds terminal icons for 250ms; any return to `loading` must apply immediately so a newly-appended tool call can take the header straight back to the spinner. Guard the delayed write with a version/token check, not cancellation alone — a `Task.sleep` that has already resumed can still commit an outdated success/error state after the group has re-entered loading unless you reject stale completions explicitly. `aggregateIsComplete` is vacuously true for empty `tools`, which would render a green check — the single-entry branch (`count <= 1` with `if let only = tools.first`) guards against this in practice, but keep the guard if you refactor the branches.
- Tool header summary text is always `.primary` — the red error `xmark.circle.fill` from `ToolStatusIndicator` is the single source of "this tool failed" in the header row. Don't re-tint the summary text red on `isError`; the icon is plenty, and tinting the semibold subheadline text red pulled visual weight away from the summary content itself.
- Expanded multi-entry groups indent their nested `InlineToolRow` list by 22pt (`chevron width 12 + HStack spacing 10`) so each child's chevron lines up beneath the header's status icon, not flush with the outer chevron. If you change the header HStack spacing or the chevron width, update this indent in lockstep.
- `TaskListBlock` status indicators:
    - **Keep checkbox slots stable.** Use fixed 16×16 slots so status changes do not reflow task rows.
    - **Scale `ProgressView` for in-progress tasks.** Render a standalone spinner scaled to visually match the checkbox glyph in the same slot instead of a custom repeat-forever SwiftUI animation, keeping animation in AppKit's progress-indicator layer and avoiding per-frame SwiftUI work while transcript rows are lazy-loaded.

### Expand/Collapse Animation

- **Single source of easing.** `toolExpansionAnimation` in `ChatBlocks.swift` is the shared `Animation` used for every bubble's expand/collapse. Tune there, not at call sites.
- **Two coordinated pieces drive the animation, both load-bearing:**
    - **`.toolAnimationOverride(value:)`** on the bubble scopes the transaction *within* the bubble's subtree so its own frame reflow eases during active turns. The outer `ChatTranscriptView` `.transaction { if turnActive { disablesAnimations = true } }` would otherwise snap the bubble during streaming; this override resets `disablesAnimations = false` and pins `animation = toolExpansionAnimation`. Pair `value:` with whatever drives the layout shape (e.g. `isExpanded`, the tool list).
    - **`withAnimation(toolExpansionAnimation) { isExpanded.toggle() }`** wraps each toggle in `StandaloneToolRow`, `ToolGroupBlock` (single-entry and multi-entry), and `InlineToolRow`. This sets the animation on the transaction *globally* for that state change, so the enclosing `LazyVStack`'s sibling positions reflow in lockstep with the bubble's shrink. **Why:** `.toolAnimationOverride` only affects the bubble's own subtree — it does not reach the parent layout. Without the `withAnimation`, the next transcript item snapped to its new position while the bubble's frame was still animating down, briefly overlapping (visually "underlapping") the collapsing bubble for the duration of the ease. Do not remove either piece on the theory that the other is sufficient; they target different scopes (within-bubble vs. surrounding layout).
- **`SubAgentBlock` applies the same pattern via a `withAnimation`-wrapping `Binding` on its `DisclosureGroup`, plus `.toolAnimationOverride(value: isExpanded)` on the bubble.** `DisclosureGroup` animates its own content reveal internally, but Apple's implementation is closed-source and not guaranteed to wrap the chevron-tap binding write in `withAnimation`, so the outer `LazyVStack` sibling reflow could snap for the same reason as the tool-row case. Routing `isExpanded` through a custom `Binding` whose `set` calls `withAnimation(toolExpansionAnimation)` keeps the block's expand/collapse consistent with the tool bubbles and defensive against the sibling-snap regression if `DisclosureGroup` doesn't already do it internally. The `.toolAnimationOverride(value: isExpanded)` mirror of the tool-bubble active-turn fallback preserves the ease when a user toggles the block during streaming (without it, the outer `disablesAnimations = true` would snap the block while tool bubbles continued to ease — an inconsistency between block types). **Tradeoff:** if `DisclosureGroup` *does* wrap internally, our explicit `withAnimation` replaces Apple's default curve with `toolExpansionAnimation` for the same update — intentional, to keep sub-agent and tool-bubble expansion timing consistent. Revisit if Apple's default starts to feel materially different in a future SwiftUI release.
- **`ToolStatusIndicator` must snap on branch swaps but animate on layout.** Use `.transaction(value: branchKey) { $0.animation = nil }` on the indicator, *not* a bare `.transaction`:
    - The `value:` scope means the animation-nil mutation fires only when `branchKey` changes (spinner ↔ checkmark ↔ xmark). That preserves the intended snap on branch transitions — a spinner appearing *as* a spinner, not fading in from an ambiguous transient state when the enclosing `toolAnimationOverride(value: tools)` or the per-toggle `withAnimation` would otherwise cross-fade it.
    - A bare `.transaction` also nulled out *layout-driven* updates, so when a neighboring bubble expanded/collapsed and the transcript's animation propagated through the `LazyVStack`, the indicator snapped to its new position while the rest of its header row eased. Value-scoping restores that parent-driven animation.
    - `branchKey` is an `Int` derived from `isError` / `isComplete`; the specific representation doesn't matter as long as any branch transition produces a different value.
