## Chat View Details

These instructions cover chat-specific view code under `Alveary/Views/Chat/`. Transcript block rendering (`ChatBlocks*`) lives in the narrower `Alveary/Views/Chat/Blocks/AGENTS.md`.

> **READ FIRST — Focus and keyboard rules are centralized.** Before touching `@FocusState`, `.onKeyPress`, or `.keyboardShortcut` on any chat surface, consult the **Focus And Keyboard Coordination** section in `Alveary/Views/AGENTS.md`. It owns modifier-shortcut placement (⌘W, ⌘1..9, etc.), why shortcut-bearing buttons must not be buried in conditionally-rendered branches, and how any new `.onKeyPress` handler would conflict with the sidebar's focus grab.

## Transcript Scrolling And Follow Mode

### Follow-Mode Mechanisms

- Transcript auto-follow should stay pinned when the user is already at the bottom and new content increases transcript height. Responsibilities are split across three mechanisms, and the split is load-bearing — do not collapse them back into a single geometry-driven path:
    - **Content-size growth**: handled by `.defaultScrollAnchor(.bottom, for: .sizeChanges)` on the `ScrollView`. This pins the bottom of the content to the viewport when size changes, and is gated on `isFollowing` so users who scrolled away are not dragged back. Covers streaming-bubble wrap, bubble expand/collapse, and any other intrinsic height change.
    - **New message / new stream chunk**: handled by explicit `scrollToBottom` calls in `onChange(of: events.count)`, `onChange(of: viewModel.messageQueue.pending.count)`, and `onChange(of: viewModel.streamingText)`. These are the only surfaces that should issue a snap-to-bottom in response to content-growth events.
    - **Container-size changes**: handled by `shouldPreserveFollowMode` inside `onScrollGeometryChange`. This covers the composer banner, error banners, and the changed-files strip appearing (shrinking the viewport) while the user was near the bottom.
        - **Only *shrinks* trigger the preserve path.** Container growth pulls the user toward the bottom on its own (more viewport, less to scroll), so there's nothing to re-pin.
        - **The offset guard is `!offsetDecreased`, not `!offsetChanged`.** `.scrollPosition(_, anchor: .bottom)` bumps `offsetY` *up* when the container shrinks to keep the content-bottom aligned to the viewport-bottom. That anchor-driven increase is not a user scroll — a previous `!offsetChanged` guard missed this and left the transcript above the bottom whenever the diff strip or an error banner arrived *after* the initial `.jumpToLatest` pending had timed out. A real user drag up decreases `offsetY`, so `!offsetDecreased` rejects user drags while allowing anchor adjustments.
- Do not re-add content growth (`contentGrew`) to `shouldPreserveFollowMode`'s condition. Bubble expand/collapse animations emit many intermediate `onScrollGeometryChange` frames; firing `scrollToBottom` on those frames causes its re-issued `scrollTo` calls to fight the bubble's own expansion animation, producing scroll jank. `.defaultScrollAnchor(.bottom, for: .sizeChanges)` is the correct mechanism for those cases.

### Layout And Scroll Modifiers

- In `ChatTranscriptView`, scroll coordination uses `scrollPosition.scrollTo(edge: .bottom)` — not `scrollTo(id:)`. Edge-based scroll asks SwiftUI to go to the content's trailing edge directly, without relying on a specific row id's position being accurate.
    - **Why**: `LazyVStack` row-height estimates are only calibrated as rows materialize. On thread re-open (navigate away then back), the grouper already has all items populated when body first runs, so `LazyVStack` must measure all rows on a single layout pass. Its initial estimate can be ~2× the real total height, and `scrollTo(id: "chat-bottom")` lands the viewport at the bottom of the over-estimate — in a range of unmaterialized rows, rendering blank until the user drags up.
    - **Cold launch works for the same reason the bug doesn't reproduce there**: the grouper is empty at first body eval, so `LazyVStack` starts at 0 rows and incrementally adds the items, calibrating heights as it goes. The edge-scroll fix makes re-open match cold-launch behavior by removing the dependency on id-based position estimates.
- **Do not apply `.scrollTargetLayout()` to the transcript's `LazyVStack`.** The modifier marks the stack's children as scroll targets, which causes `.scrollPosition(anchor: .bottom)` and `.defaultScrollAnchor(.bottom, ...)` to align "last scroll target's bottom = viewport bottom". That alignment ignores the outer `.padding(.bottom, ...)` (which lives outside the scroll targets) and silently drifts `offsetY` up by exactly 14pt a few frames after a scroll lands — no `onScrollGeometryChange` tick is emitted for the drift. The user-visible symptom is a partially cut-off bottom inset on thread re-select. `.scrollTargetLayout()` is only needed when pairing with `scrollTargetBehavior(.viewAligned)` (snap-to-element paging), which we don't use. Leaving the modifier off makes all three SwiftUI scroll-anchor mechanisms agree on the same content-end bottom.
- The bottom inset (`transcriptBottomInset` = 14pt) is applied as trailing `.padding(.bottom, ...)` on the `LazyVStack` — **not** as an inner `Color.clear` trailing child.
    - **Do not move it inside the `LazyVStack` as a clear child row.** A trailing child is subject to the stack's *estimated* row positions during a single-pass layout on thread re-open (when the grouper has all items pre-populated, `LazyVStack` measures everything at once and the estimate can be wildly off for heavy content like markdown tables). `scrollTo(edge: .bottom)` then anchors to an inflated bottom and the viewport lands in an unmaterialized range, rendering the transcript **blank until the user drags up**. Outer padding is a fixed addition that isn't subject to the stack's row estimation, so edge-scroll lands reliably.

### `isFollowing` And `nextFollowingState`

- Treat content-size growth differently from a user-initiated scroll-away so the `Jump to bottom` affordance only appears after the user actually leaves the bottom. `isFollowing` is the source of truth; the fallback branch of `onScrollGeometryChange` routes through `ChatTranscriptScrollBehavior.nextFollowingState(currentIsFollowing:oldMetrics:newMetrics:)`, **not** a raw `isFollowing = newMetrics.isNearBottom` write.
    - **Why**: during streaming a single markdown chunk can mount a ~100pt+ block on one render pass. SwiftUI fires `onScrollGeometryChange` with the new `contentHeight` *before* `.defaultScrollAnchor(.bottom, for: .sizeChanges)` bumps `offsetY` to re-pin — so `distanceFromBottom` momentarily spikes past the 60pt `isNearBottom` threshold even though the user didn't scroll.
    - **What the raw write broke**: flipping `isFollowing` to `false` on that transient tick disarmed `.defaultScrollAnchor(isFollowing ? .bottom : nil, for: .sizeChanges)` on the very next size change, so subsequent content growth silently extended below the viewport; `onChange(of: streamingText)`'s `guard isFollowing` also short-circuited, blocking follow-up `scrollToBottom` corrections. Symptom: the jump-to-latest button blinked during streaming and then stuck visible at turn end.
    - **The rule `nextFollowingState` enforces**: set to `true` when `isNearBottom` (anchor has caught up), set to `false` only when `shouldCancelProgrammaticScroll` (actual user drag — offsetY *decreased* AND distance grew), otherwise leave `isFollowing` alone. Keeps follow mode sticky through transient layout lag while still respecting user intent.

### Retry Strategies And Progressive Scroll

- `scrollToBottom` has two retry strategies, selected via its `retries` parameter:
    - **`.triple`** (default): immediate `scrollTo` + next-runloop + 150ms retries, plus the `transcriptProgrammaticScrollTimeout` cleanup. Used for `jumpToLatest` (thread entry, where async composer layout can shift the bottom after the initial scroll lands) and for the content-growth preserve-follow callers (`events.count`, `messageQueue.pending.count`, `streamingText`) where async bubble layout may still shift after the initial scroll.
    - **`.single`**: immediate `scrollTo` only + `transcriptProgrammaticScrollTimeout` cleanup. Used by the container-change preserve-follow path in `onScrollGeometryChange`. Continued container animation is re-pinned by `shouldReissuePendingPreserveFollow` on each subsequent geometry tick, so the deferred retries would be redundant work and the initial triple-fire would amplify per-frame container changes (e.g. composer banner animating in over 300ms).
- **`forceFollow` scrolls from far-up use a progressive stepped fallback** (`performProgressiveScrollToBottom`). SwiftUI's `scrollPosition.scrollTo(edge: .bottom)` effectively jumps the viewport to the target offset in a single layout pass; `LazyVStack` only materializes rows intersecting the *final* viewport, so if the user was scrolled hundreds of points from bottom (e.g. top of a long thread) then clicked "jump to bottom", the bottom rows may never materialize and the transcript renders blank until the user drags. The progressive path sidesteps this by issuing a series of `scrollPosition.scrollTo(y:)` calls in 300pt increments with ~40ms between each, so the viewport actually sweeps through intermediate positions and `LazyVStack` materializes the rows along the way. Rules:
    - **Gate on `forceFollow && distance > 400pt && content > viewport`.** Near-bottom `forceFollow` scrolls (turn-end rebuild, streaming, thread entry where `.defaultScrollAnchor(.bottom)` already puts us near the bottom) take the fast single `scrollTo(edge: .bottom)` path. Only long jumps need stepping.
    - **The progressive scheduler owns the scroll flow while it runs.** Gate on `isProgressiveScrolling` in the in-pending `.reissue` branch and skip the triple-retry ladder's `scrollTo(edge: .bottom)` firings — running both simultaneously causes them to race each other and `LazyVStack` never gets a stable frame to materialize. The scheduler itself calls `scrollTo(edge: .bottom)` for its final hop so the existing reissue predicates can still converge to the exact bottom.
    - **Refresh the watchdog on every progressive step.** `schedulePendingProgrammaticScrollTimeout()` must be called inside `performProgressiveScrollToBottom` each iteration — and also from the `.reissue` branch in `pendingScrollAction` even when its scrollTo is suppressed. The reissue predicate only fires on container-shrink / content-grow ticks; during a progressive sequence where `LazyVStack` has already stabilized its contentHeight, intermediate ticks resolve to `.noop` (which never refreshes the watchdog). Without explicit refresh from the scheduler, the watchdog could fire 400ms in and clear `pendingProgrammaticScrollMode`, causing the next step's `guard` to exit the chain partway through and strand the transcript mid-scroll.
    - **Reset `isProgressiveScrolling = false`** on every path that ends the sequence — the final-edge hop, the pending-was-cleared early exit, and the metrics-nil fallback. Leaking the flag would persistently suppress reissues for the lifetime of the view.
    - **Cap the recursion with `transcriptProgressiveScrollMaxSteps`.** The loop normally terminates when `distance <= step`, but if content keeps growing faster than we step or the scroll refuses to advance, the guard would never fire. The step count caps total scroll travel at ~9000pt, far beyond any realistic transcript, and bails to a final `scrollTo(edge: .bottom)` when the cap is hit. Without this, a degenerate case could loop indefinitely — pending won't clear via the watchdog because the reissue branch keeps refreshing it on each step's geometry tick.
    - **Guard against concurrent chains with `!isProgressiveScrolling`.** Rapid repeat `forceFollow` calls (double-tap the jump-to-latest button, send-while-mid-progressive) must not start a second chain alongside the first — two chains would race each other's `scrollTo(y:)` calls and `LazyVStack` would never settle. The in-flight chain already reads `latestMetrics` on every step, so it naturally adapts to content growth that happens after it started; a second chain would just be redundant. The retry ladder's gate uses `!isProgressiveScrolling` (not `!usingProgressiveScroll`) for the same reason: a repeat call whose `usingProgressiveScroll` evaluates false (because we skipped starting a second chain) would otherwise fall through to the retry ladder's `scrollTo(edge: .bottom)` while the existing progressive chain is still driving.

### Pending-Scroll Watchdog

- The pending-scroll timeout (`transcriptProgrammaticScrollTimeout`, 400ms) is a **reset-on-progress watchdog**, not a fixed deadline from the kickoff:
    - **Why**: entering a large existing thread, `LazyVStack` can take longer than 400ms to materialize enough rows to produce a stable content size. A one-shot deadline from the initial `scrollTo` would fire while content was still growing, clear `pendingProgrammaticScrollMode`, and stop the `shouldReissuePendingJumpToLatest` reissue loop — leaving the viewport above the true bottom and the transcript rendering blank until the user dragged up.
    - **How**: `schedulePendingProgrammaticScrollTimeout()` stamps a fresh `pendingProgrammaticScrollTimeoutToken` (UUID) each time it's called; the scheduled `asyncAfter` closure only runs if its captured token still matches. Call it from the initial `scrollToBottom` and from every reissue branch in `onScrollGeometryChange` (both `jumpToLatest` and `preserveFollow`). The watchdog fires 400ms after the *last* reissue, so the pending mode only clears once content has settled (or the user has cancelled via `shouldCancelProgrammaticScroll`).
    - **Do not overwrite `isFollowing` in the watchdog.** An earlier iteration wrote `isFollowing = latestMetrics?.isNearBottom ?? false` in the `.jumpToLatest` branch on watchdog fire. On app launch to a preselected thread, `onScrollGeometryChange` had not yet emitted a tick when the watchdog landed — `latestMetrics` was nil, `?? false` flipped `isFollowing` to `false`, and the jump-to-latest button flashed for a frame until a later geometry tick restored it through `nextFollowingState`. `isFollowing` was set to `true` at kickoff; `.cancelled` is the only legitimate path to `false` during the pending window, and it clears the pending mode early enough that the watchdog's guard short-circuits. The watchdog's only job is to clear `pendingProgrammaticScrollMode`.
    - **Do not issue a corrective `scrollTo` from the watchdog either.** A briefly-tried "final catch-up" in the watchdog (scrollTo(edge: .bottom) if near-but-not-at bottom before clearing pending) regressed app launch to large threads: the explicit `scrollPosition.scrollTo(...)` inside the `asyncAfter` closure interacted badly with `.defaultScrollAnchor(.bottom, for: .sizeChanges)` during `LazyVStack` calibration, leaving the transcript scrolled well above the bottom with the jump-to-latest button showing.

### `.jumpToLatest` Pending-Mode Specifics

- Programmatic `jumpToLatest` scroll on thread entry is not a one-shot:
    - Composer-area content that loads asynchronously (e.g. the changed-files strip once `DiffViewerViewModel.files` populates) can shrink the transcript viewport *after* `scrollTo` lands, leaving the transcript above the new bottom.
    - While the pending scroll is live, container-height shrinks or content growth must re-issue `scrollPosition.scrollTo(edge: .bottom)` via `shouldReissuePendingJumpToLatest` and refresh the watchdog timeout. This recovery path *does* react to content growth, unlike `shouldReissuePendingPreserveFollow`: the pending-jump window only runs during thread entry / turn-end rebuilds, so it will not interact with per-bubble expand animations from normal transcript use.
- **`.jumpToLatest` does NOT clear its pending mode on a transient `isAtBottom`.** On a large existing thread, `LazyVStack` initially materializes a small window of rows near the bottom anchor. At that moment `isAtBottom` is trivially true (content height equals viewport height or less), so a naive "clear on `isAtBottom`" would disarm the reissue loop immediately. But `LazyVStack` then replaces row-height *estimates* with actual measurements as more rows materialize — the reported `contentHeight` shifts, our scroll offset stays at the old bottom, and the viewport ends up above the real content end. Scrolling up pulls lazy rows into the viewport, which is why the user-visible symptom is "transcript blank until I scroll, then content appears." The `isAtBottom` branch therefore only clears pending mode for `.preserveFollow`; `.jumpToLatest` stays pending until the reset-on-progress watchdog fires after 400ms of no further content growth. `isFollowing` is still set to `true` immediately so the jump-to-latest button doesn't flash.

### `.preserveFollow` Pending-Mode Specifics

- `.preserveFollow` pending scrolls (kicked off by a container-size change) also participate in pending-window re-issuance, via `shouldReissuePendingPreserveFollow`:
    - Only container *shrinks* re-issue. Content growth is excluded: `.defaultScrollAnchor(.bottom, for: .sizeChanges)` handles content growth while `isFollowing` is true, and re-issuing on every streaming or bubble-expand frame would re-introduce the jank the preserve-follow narrowing was meant to fix.
    - Container *growth* during a pending preserve-follow scroll is also excluded — a growing viewport pulls the user toward the bottom on its own, and `isAtBottom` will clear the pending mode naturally.

### Cancel Predicate Guards

- User intent still wins: evaluate `shouldCancelProgrammaticScroll` before re-issuing, so a real user drag up during the pending window cancels the jump instead of fighting it. **Four** conditions must all hold — each rejects a distinct SwiftUI-initiated false positive:
    - **offsetDecreased**, not merely changed. `.defaultScrollAnchor`'s normal catch-up during streaming *increases* offsetY toward the bottom; only a real user drag up decreases it.
    - **distanceGrew**, so a scroll converging on the bottom never trips the cancel.
    - **`!newMetrics.isNearBottom`** — the new position must be clearly past the 60pt near-bottom band. This rejects the streaming anchor-adjustment false positive: when content fits the viewport, `.scrollPosition(anchor: .bottom)` pins content using *negative* offsetY, and a `scrollTo(edge: .bottom)` can swing offsetY by hundreds of points (e.g. 0 → -377) while landing the new distance within `isNearBottom` (e.g. 12pt). Without this guard, every streaming chunk could false-fire cancel and briefly flip `isFollowing` to false, flashing the jump-to-latest button mid-stream.
    - **offsetDrop < `transcriptCancelMaxPerTickDrop`** (250pt) — the offset drop must be within a plausible per-tick user-drag magnitude. This rejects the turn-end false positive: `forceFullRebuild` regenerates tool-group UUIDs, SwiftUI's ForEach diffs as heavy remove+insert, `.defaultScrollAnchor` loses its anchor view mid-diff, and the ScrollView snaps offsetY to a stale value in a single geometry tick (observed: a 347pt drop in 1ms, from `off=893 dist=0` to `off=546 dist=347`). `distance=347` is past the near-bottom guard, so that guard alone would not catch this case — the velocity cap does. The 250pt threshold is loose enough to admit very fast trackpad flings (the existing "user drag-away" test uses a 160pt drop) while rejecting multi-hundred-pt programmatic snaps. Don't tighten it below that band or real user drags will be mis-classified as rebuild disturbances.

### Turn-End Re-Pin

- **Turn end must re-pin the transcript when the user was following.** `onChange(of: viewModel.turnState.isActive)`'s false branch calls `viewModel.rebuildChatItemsIfNeeded(from: events, forceFullRebuild: true)` and must then re-anchor the scroll:
    - **Why**: the `forceFullRebuild` regenerates tool-group identities with new UUIDs, so SwiftUI's ForEach diffs the list as heavy remove+insert. Simultaneously, the streaming bubble (`id("streaming")`) unmounts as `streamingText` goes nil. `.defaultScrollAnchor(.bottom, for: .sizeChanges)` loses the view it was pinning to mid-diff, and the ScrollView falls back to a stale offset (the one from before streaming started).
    - **How to apply**: after the rebuild, call `scrollToBottom(forceFollow: true)` when `isFollowing`. Skip the re-pin when `isFollowing == false` so the user's deliberate scroll-away is respected.
    - **Use `forceFollow: true` (jumpToLatest), not preserveFollow**: the triple-fire retries and `shouldReissuePendingJumpToLatest`'s content-growth reissue window catch the rebuild's layout shifts; preserveFollow's container-only reissue predicate does not.

## Bubble Content Rendering

- `EmptyThreadState` checks `isCancellingInitialSetup` before `setupPhase` so cancellation feedback takes precedence even when `setupPhase` is still set mid-rollback. Keep that ordering if you restructure the view; otherwise the empty-thread pane will flicker back to "Creating worktree" during the rollback shell commands.
- `UserBubble` routes *all* rendered text through `AppMarkdownText` — do not reintroduce a plain-`Text` fast path for messages without backticks, even though it means messages like `"I love *these* edits"` now render the italic markdown emphasis. That's intentional: transcripts should render as markdown. The markdown pipeline also attaches slash-command and `@`-mention chips via `AppMarkdownParser.attachComposerChips(to:)`, reusing `ChatInputFieldTextSupport.composerTextChips(in:)` for detection. Chip labels are always the path's `lastPathComponent`, so the bubble does *not* need a working directory to render correctly — if a future design needs per-thread relative paths or ancestor trimming, restore a `workingDirectory` parameter on `composerTextChips` and `mentionChipDisplayText`, and thread the thread's `worktreePath ?? project.path` through `ChatView` → `ChatTranscriptView` → `UserBubble` at that point.
- `attachComposerChips(to:)` skips any attributed-string range that already carries a Textual attachment (images/emoji resolved by `WithAttachments`), a markdown `.link`, a `.codeBlock` block-level `presentationIntent` (fenced code block), or a `.code` `inlinePresentationIntent` (backtick inline code). The inline-code guard is load-bearing: `composerTextChips` is invoked here with the *parsed* flat string (backticks already stripped), so the helper's own `codeRanges`-based filter returns nothing to exclude. Without the guard, a user writing `` `@path/to/file.swift` `` in a message would have their inline code clobbered by a composer chip that truncates the path to just `@file.swift`. Keep all four conditions together — each covers a distinct case that the others don't.
- Transcript markdown links like `[docs](Alveary/DI/AGENTS.md)` or `[shot](~/Desktop/file.png)` open via an `.environment(\.openURL, ...)` modifier on `ChatTranscriptView`, not SwiftUI's default. The modifier sits on the outer transcript view so both `UserBubble` and `AssistantBubble` inherit it — do not re-scope it to a single bubble type.
    - **Resolve schemeless URLs against `workingDirectory`.** Foundation's markdown parser hands relative links to `openURL` as schemeless `URL` values; SwiftUI's default handler passes them straight to `NSWorkspace.shared.open(_:)`, which silently no-ops without a `file://` prefix. `ChatTranscriptView.resolveMarkdownLinkURL(_:workingDirectory:)` rebuilds the URL against `URL(fileURLWithPath: workingDirectory, isDirectory: true)` so the click opens the file.
    - **Expand `~` before the working-directory branch.** Tilde is a shell convention, not a URL convention — `URL` treats `~/Desktop/foo` as a plain relative path and naive baseURL resolution produces a bogus `.../alveary/~/Desktop/foo`. The resolver detects the `~` prefix first, calls `removingPercentEncoding` (Foundation's markdown parser percent-encodes path chars, and `expandingTildeInPath` operates literally — without the decode, `~/Desktop/my%20file.png` lands as a `my%20file.png` filesystem lookup and misses the real `my file.png`), runs `NSString.expandingTildeInPath`, and wraps the result with `URL(fileURLWithPath:)`. Keep this branch ahead of the workingDirectory branch; otherwise tilde links open nothing.
    - **Pass absolute URLs through unchanged.** Any URL with a scheme (`https`, `file`, `mailto`, etc.) must reach `NSWorkspace.shared.open(_:)` as-is — the resolver only touches `scheme == nil` inputs so external links keep working.
    - **Pass fragment-only references through unchanged.** A link like `[top](#section)` is schemeless with no path; naively resolving it against `workingDirectory` yields `file:///.../cwd/#section` and opens the cwd in Finder. Guard on `hasPrefix("#")` before the tilde branch so `NSWorkspace.shared.open(_:)` silently no-ops instead.
    - **Keep `workingDirectory` plumbed from `ChatView`.** `ChatView` receives `conversation.thread?.worktreePath ?? conversation.thread?.project?.path` via `ConversationView.activeWorkingDirectory`; forward that into `ChatTranscriptView(workingDirectory:)` so per-thread worktree links resolve correctly. `ChatTranscriptLinkResolutionTests` pins each branch (scheme pass-through, fragment-only, tilde with and without percent-encoding, workingDirectory resolution for plain/percent-encoded/path-absolute inputs, nil/empty workingDirectory) — update it in lockstep if the resolver format changes.
    - **File-mention chips in `UserBubble` are clickable and reuse the same `openURL` handler.** `AppMarkdownParser.applyComposerChip` tags every `.fileMention` chip's replacement run with `replacement.link`, so SwiftUI routes clicks through the transcript's `OpenURLAction` alongside regular markdown links. Absolute stored paths (including tilde-expanded) become `file://` URLs via `URL(fileURLWithPath:)`; relative stored paths stay schemeless so `resolveMarkdownLinkURL` resolves them against the thread's `workingDirectory` at click time. Do not add a separate per-bubble click handler — the whole point is to share the schemeless/tilde/fragment logic already tested in `ChatTranscriptLinkResolutionTests`. Slash-command chips intentionally stay unlinked (purely visual).

## Composer And Tabs

- Composer top separators that appear while the transcript is scrolled up should live in the composer panel's background layer, not a parent overlay, so autocomplete popups can cover them without the separator bleeding through.
- Conversation rename in multi-conversation tabs is inline via `editingConversationID` in `ConversationTabChip`, not a separate modal flow. The editing variant renders a distinct capsule fill (`NSColor.textBackgroundColor` + 1pt accent stroke, matching macOS Finder inline-rename) and suppresses the `×` close button via `.tabChipShell(..., showsCloseButton: false)` so users aren't left guessing whether the `×` commits, cancels, or deletes. The shell still reserves the trailing geometry so chip width stays constant when toggling between display and rename modes. The outer `.contextMenu` and the rename VoiceOver rotor action (via `SelectableTabChip`'s `renameAccessibilityAction:`) are both gated on `editingConversationID == nil` — forbidding a rename on *any* tab while another is in-flight — because swapping `editingConversationID` mid-edit left the target chip stuck in editing state without an input field (the simultaneous unmount/mount of TextFields in a single SwiftUI update pass didn't converge). The sidebar thread row applies the same guard for the same reason.
- `ThreadDetailConversationTabs` should keep the system `.bar` background for the header chrome and add any custom separator as an overlay. Replacing the bar with `windowBackgroundColor` creates an unintended dark strip in the live app.
- Conversation tab chips are not list rows, but they should mirror the same press-feedback principles: let the select action own the full capsule hit area, overlay trailing affordances like the close button on top of that surface, and prefer fill changes over capsule strokes for selected styling because macOS can render stray vertical artifacts from chip outlines in snapshots.
- Conversation tabs render through the shared `SelectableTabChip` (`Alveary/Views/Components/SelectableTabChip.swift`), which routes its label through `AppMarkdownInlineLabel` — chip fill is always `.standard` so the color stays uniform across selection transitions. Terminal session chips share the same shell. Do not hand-roll a separate status-dot + label + close-button capsule for another tab-like surface — extend `SelectableTabChip` (or layer modifiers around it) so the surfaces cannot drift on padding, accessibility traits, or chip-color rules.
- Conversation-tab shortcuts live in `ThreadDetailView+ConversationTabs.swift`:
    - ⌘1..⌘9 attach to each chip's visible select `Button` via the optional `KeyboardShortcut?` variant, scoped to the multi-conversation branch (single-conversation headers render a label, so no binding).
    - ⌘W does *not* ride on the per-chip close `Button`; per-chip close bindings didn't reliably override the system "Close Window" when the first chip was selected. Instead, one invisible `Button` attached as a `.background` to the tab bar carries `⌘W` and calls `onRemove(selectedConversation)`.
    - The ⌘W button must be mounted unconditionally (not gated on `conversations.count > 1`) and guard internally: during an inline rename *or* when there is only one conversation, the action returns early. Closing a thread is not a concept, so ⌘W must absorb the key in those cases rather than fall through to the scene's default "Close Window" and kill the app window.
    - Keep the ⌘W button attached via `.background`, not as an HStack sibling. A zero-framed child of the tab-bar HStack still consumes the HStack's `spacing`, shifting the visible layout and breaking snapshots.
    - The ⌘W button must carry `.id(selectedConversation.persistentModelID)`. Without a selection-scoped identity SwiftUI keeps the shortcut's bound action captured from the first mount, so ⌘W always closes the first tab even after selecting another.
    - Use an enabled, no-op action rather than `.disabled` for the guarded cases. A disabled shortcut-bearing button lets ⌘W fall through to the scene's default "Close Window", whereas an enabled no-op absorbs the key press.
    - Keep everything off of `.commands` / `CommandGroup`. Those always surface in the menu bar and can be preempted by the scene's default close handling.
    - When `removeConversation` deletes the currently-selected tab, pick the visual neighbor (next, falling back to previous) before the delete/save and set it via `appState.selectConversation`. `repairSelectedConversationIfNeeded` otherwise falls back to the main conversation via `selectedConversation(in:)`'s main-preference sort, which jumps selection back to the first tab instead of the adjacent one when the user closes a non-first tab.
    - `ThreadDetailView`'s `onRemove` handler must re-check `conversations.count > 1` before setting `pendingDeleteConversation`. The child ⌘W button's in-closure guard alone is not sufficient: after back-to-back deletions down to a single tab, ⌘W can still trip the confirmation dialog, and `removeConversation`'s backend guard then surfaces an error banner. The parent-level guard reads fresh model state via the computed `conversations` property (it goes through `thread.conversations` on the live SwiftData relationship), so it catches the case regardless of the underlying SwiftUI diffing behavior.

- **Capture both `persistentModelID` and the UUID-string `id` synchronously inside the confirmation dialog's `Remove` button closure, and pass both into `removeConversation(id:conversationIDString:)`.** Do not let the async helper re-resolve a `Conversation` via `modelContext.model(for:)` only to read `.id` off it — `model(for:)` can return a zombie reference whose first persisted-property access traps with `_assertionFailure`. See **Async Property Access** in `Alveary/Data/AGENTS.md` for the full rule and why pre-await capture is safer than re-fetching.

## Conversation Tab Row: Scroll Hooks And Sentinel

- **Wrap the multi-conversation `ScrollView` in a `ScrollViewReader` and tag each chip with `.id(conversation.persistentModelID)`.** Mirrors the terminal-pane rule: without an explicit `.id` per chip, `ScrollViewProxy.scrollTo` has no target to register against.
- **The trailing sentinel view (`Color.clear.frame(width: 12)` with `.id(ScrollTarget.trailingSentinel)`) does double duty — it reserves the 12pt visible breathing gap between the last chip and the overlay divider AND serves as the scroll target whose trailing edge is the content's absolute trailing edge.** Required because `proxy.scrollTo(lastChipID)` with the default `nil` anchor does minimum-scroll-to-visible and stops when the chip's trailing meets the viewport's trailing — a separate `.padding(.trailing, 12)` would then sit offscreen and the last chip visually butts the divider (observed with ⌘T on a full tab row). With the sentinel *as* the gap, scrolling to it with `anchor: .trailing` lands `contentOffset == tabsMaxScrollableDistance` and the full 12pt stays on-screen.
    - **Structure:** wrap the chip HStack inside an outer `HStack(spacing: 0)` and place the sentinel as the outer HStack's final element. Do not put the sentinel inside the inner (spacing: 6) HStack — spacing would push the sentinel inward and its trailing edge would stop 6pt short of the content trailing edge, reintroducing the clip-at-divider bug.
    - **Do not reintroduce a separate `.padding(.trailing, 12)` on the chip HStack.** The sentinel's 12pt width already reserves the gap; a second padding would double the visible gap to 24pt at scroll-to-end.
- **Scroll on selection via `onChange(of: selectedConversation.persistentModelID, initial: true)`. If the selected ID matches `conversations.last?.persistentModelID`, target `ScrollTarget.trailingSentinel` with `anchor: .trailing`; otherwise target the chip ID with the default `nil` anchor.** Mid-row selections still use minimum-scroll-to-visible so already-visible chips don't jump; last-chip selections route through the sentinel so the 12pt trailing gap is preserved. `initial: true` covers the first render.
- **Also scroll on count-grow via `onChange(of: conversations.count)` guarded on `newCount > oldCount`, targeting `ScrollTarget.trailingSentinel` with `anchor: .trailing`.** `createConversation` appends, so any count-grow should surface the newly-added (last) conversation — the sentinel guarantees the row scrolls all the way to the end. The guard skips the removal path (count decreases). This mirrors the terminal pane's `sessions.count` hook in intent, but the conversation row adds the sentinel step because it has a trailing gap the terminal row doesn't.

## Conversation Tab Row: Divider Mechanics

- **Only the trailing edge divider exists for conversation tabs — no leading divider.** Unlike the terminal pane (which has a fixed terminal icon to the left of the scrolling row), the conversation row starts flush against the pane edge, so a leading divider would have no fixed element to abut. The trailing divider sits between the scrolling tabs and the pinned `New Conversation` button and hides once the row is scrolled all the way to the end.
- **Drive the divider off `onScrollGeometryChange`, not a `GeometryReader + PreferenceKey`.** `ConversationTabsScrollGeometry` snapshots `contentSize.width`, `containerSize.width`, and `contentOffset.x`; `hasTabsBehindTrailingEdge` gates the overlay on `effectiveMaxScroll > 0.5 && contentOffset < effectiveMaxScroll - 0.5`, where `effectiveMaxScroll = tabsMaxScrollableDistance - tabsTrailingSentinelWidth`. The subtraction is load-bearing: the sentinel adds its 12pt to `contentWidth` but is not a chip, so the last 12pt of scroll is reserved trailing space. Without the subtraction the divider would keep rendering while only the sentinel scrolled into view (no chip clipped), and would draw a false-positive divider when chips nearly fit the viewport and only the sentinel overflows. Share the same tint (`Color.primary.opacity(colorScheme == .dark ? 0.35 : 0.3)`) and 18pt height as the terminal pane so the two surfaces do not drift visually.
- **Render the divider as `.overlay(alignment: .trailing)` on the ScrollView.** Overlays don't affect the ScrollView's own layout, so the divider appears / disappears without shifting the tab content by 1pt. An inline HStack sibling moves the divider out of the ScrollView's frame, which is not what we want: the user has explicitly asked for the space left of the divider to live *inside the scrollable content*, not in an external gutter.
    - **Do not try to reserve space left of the divider via `.contentMargins(.trailing, N, for: .scrollContent)`.** On macOS 26 it does not reserve visible trailing space in this layout (repro: chip fill measured 0.5pt from divider in the overflow snapshot when contentMargins was used). The trailing sentinel view (see the **Scroll Hooks And Sentinel** section above) is the working alternative.

## Conversation Tab Row: Outer Layout And Regression Guard

- **Keep the ScrollView greedy (`.frame(maxWidth: .infinity)`) in the multi-conversation branch, and render `Spacer()` only in the single-conversation label branch.** The `New Conversation` button right-aligns because the ScrollView consumes the remaining width and the button carries `.padding(.leading, 12)`; do not reintroduce a sibling `Spacer()` alongside a flexible ScrollView or the trailing divider will float mid-row instead of hugging the button.
- **The 20pt pane-edge inset lives *inside* the scrollable content, not on the outer tab-row.** The outer row uses `.padding(.trailing, 20)` only; the multi-conversation HStack carries `.padding(.leading, 20)` so the first chip appears 20pt in at `contentOffset == 0` while letting chips scroll under the pane's visible leading edge. The single-conversation label branch applies `.padding(.leading, 20)` to the label directly so the text still sits 20pt from the pane edge.
    - **Why:** the user explicitly asked for chips to be able to scroll past the pane's visible leading edge — chips should clip against the pane boundary, not a 20pt inner boundary. Moving the inset inward preserves the 20pt visual at `contentOffset == 0`.
- **The 12pt left of the divider is the trailing sentinel's width, not a reserved scroll region.** The sentinel (see the **Scroll Hooks And Sentinel** section above) sits at the content's trailing edge and is 12pt wide, so at scroll-to-end there's a 12pt gap between the last chip and the overlay divider. Mid-scroll, any chip being clipped by the trailing edge sits flush against the divider — that clip-flush behavior is intentional: the gap lives inside the scrollable content and only shows when the content's natural trailing reaches the visible area. Pair with `.padding(.leading, 12)` on the `New Conversation` button for 12pt of breathing room right of the divider (outside the scrollable content).
    - **Why:** the user explicitly wanted the space before the divider to be *inside the scrollable content* rather than reserved in a non-scrollable band. A sibling divider or `contentMargins`-reserved band would put the space outside the scroll area — visually similar at scroll-to-end, but conceptually wrong and produces a permanent gap at mid-scroll that the user rejected.
- **`testConversationTabsOverflow` is the regression guard for the greedy-ScrollView layout, not the divider.** 8 tabs at 500pt width force the overflow state, which pins the flexible ScrollView + trailing `New Conversation` button layout — reintroducing a sibling `Spacer()` alongside the flexible ScrollView (for instance) would shift the button position and fail verification.
    - **The trailing-edge divider is NOT captured in this baseline.** `onScrollGeometryChange` dispatches its action asynchronously on a later runloop tick, after the snapshot pass's `displayIfNeeded()`, so `hasTabsBehindTrailingEdge` is still `false` at capture time and the overlay doesn't render. Regressions to `overlay(alignment: .trailing)` / `onScrollGeometryChange` / `hasTabsBehindTrailingEdge` will **not** be caught by this test — verify the divider appears at initial scroll offset and hides once scrolled to the end manually in the running app. The terminal pane's `testTerminalPaneSessionsOverflow` *does* capture its equivalent divider (different layout timing stabilizes the geometry before the displayIfNeeded boundary), so don't assume parity between the two.

## Interaction Contracts

These capture conversation-view interaction patterns. Keep new UI aligned with them unless you are intentionally redesigning the behavior across the app.

- Session reconfiguration is a between-turn action. Do not let agent/session setting changes reconfigure a conversation while a turn is active or a send is still in flight; those changes must wait until the current turn finishes.
- Composer-dropdown `apply*Change` handlers live on `ConversationViewModel` in `ConversationViewModel+Settings.swift`. The three fork-triggering handlers (`applyModelChange` / `applyEffortChange` / `applyPermissionModeChange`) each run their state/DB write synchronously and return a `@discardableResult Task<Void, Never>` carrying the async fork; `applyWorktreePreferenceChange` is a plain `Void`-returning DB write (no fork, only editable before first send) and lives in the same file. Rules when editing any of them:
    - **Do not inline the handler logic back into `ChatView`.** The view-model home is what makes them unit-testable against a `MockAgentsManager`; the regression suite in `ConversationViewModelTests+Settings.swift` depends on that entry point.
    - **Call the handlers directly from `Picker` `set:`.** No outer `Task { await ... }` wrapper in the binding. The sync prologue must run on the same cycle as the click so SwiftUI's next render observes the new value; an outer `Task` defers the prologue by one MainActor cycle and briefly paints the stale selection.
    - **Every handler must start with `guard canApplySettingsChange else { return ... }`.** That rejects writes while `turnState.isActive` or `isSendingMessage`. The composer's `.disabled(areControlsDisabled)` is a UI-level block; this is the defense-in-depth gate against stray binding writes (programmatic, races on mode flips) silently persisting a new setting to the DB.
    - **Do *not* gate the fork on `agentsManager.isRunning(conversationId:)`.** Claude's `-p --input-format stream-json` process can exit between turns, so an `isRunning` check silently drops the fork and the user sees no feedback even though the thread has a session to fork from. `reconfigureSession` already handles a dead process correctly (no-op teardown, then `--resume <id> --fork-session` spawn). Gate through `shouldReconfigureOnSettingChange()` — "thread completed initial setup" — which is the real precondition.
    - **Do *not* add a `!isReconfiguringSession` check at the handler layer.** Concurrent fork attempts are handled inside `reconfigureSession` itself (`!state.isReconfiguringSession` guard silently returns), and the composer enters `.progressOnly(.reconfiguringSession)` mode which disables the pickers while a fork is in flight.
- Queued messages stay stacked above the chat composer until they are actually sent. Do not render pending queued entries in the transcript as if they were already part of the conversation history.
- Once a queued message is actually attempted, it belongs to the transcript. If that attempted send fails, show retry affordances on the transcript user message rather than moving it back into the queued-message list.
- User-requested turn cancellation is an interruption, not a generic failure. Stopped turns should clear composer error banners, render a centered `Interrupted` transcript note, and persist a `stop` session note so restore/archive context does not summarize the turn as an error.
- User-requested cancellation of initial-setup (worktree creation + agent spawn for a new thread) is a reset, not a failure. `ConversationViewModel.cancel()` must cancel the tracked `initialSetupTask` and flip `state.isCancellingInitialSetup = true` so the composer shows a spinner instead of the stop button; the existing `rollbackFailedInitialSetup` path restores the draft and clears `hasCompletedInitialSetup`, and `sendDraft` / `retryDraft` must swallow `CancellationError` so no error banner appears. A subsequent send re-enters setup via the normal `needsSetup` check.
- While a turn is active, keep transcript updates incremental. Persisted live-turn events should append directly into `ChatItemGrouper`, and full transcript regrouping from the `events` query should be deferred until the turn ends so the active turn does not starve composer interactions like autocomplete or text insertion.
- Live root-assistant `messageChunk` events should be coalesced before they hop onto the main actor. Do not process every streamed text delta as its own `MainActor` mutation, or active turns can starve transcript completion and composer interactions.
- `ConversationViewModel` agent subscriptions are view-lifecycle owned, not initializer-owned. Keep `activateViewLifecycle()` / `deactivateViewLifecycle()` wired from `ConversationView`'s `.task` and `.onDisappear` instead of restarting subscriptions from `init`, because parent SwiftUI refreshes can recreate the model and churn `activeSubscriptionToken`.
