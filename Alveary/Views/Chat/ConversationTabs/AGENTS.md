## Conversation Tab Row

Rules for `ThreadDetailView+ConversationTabs.swift` — chip rendering, inline rename, keyboard shortcuts, scroll hooks, sentinel view, and trailing divider for the conversation tab row at the top of a thread pane.

> **READ FIRST — Focus and keyboard rules are centralized.** Modifier-shortcut placement (⌘W, ⌘1..9) is discussed below for this specific surface, but the cross-surface patterns live in the **Focus And Keyboard Coordination** section of `Alveary/Views/AGENTS.md`. Consult that section before touching any new `@FocusState`, `.onKeyPress`, or `.keyboardShortcut`.

## Chip Rendering And Rename

- Conversation tabs render through the shared `SelectableTabChip` (`Alveary/Views/Components/SelectableTabChip.swift`). Chip fill is always `.standard` so color stays uniform across selection transitions. Terminal session chips share the same shell. Do not hand-roll a separate status-dot + label + close-button capsule — extend `SelectableTabChip` so the surfaces can't drift on padding, accessibility traits, or chip-color rules.
- Conversation tabs use the shared tab-chip status slot for both dots and the busy spinner. Keep `.busy` on `TabChipStatusIndicator.spinner(.blue)` in the same fixed `8×8` frame as the dot so the chip height does not jump when a conversation starts or stops working.
- Conversation rename in multi-conversation tabs is inline via `editingConversationID` in `ConversationTabChip`, not a separate modal flow.
    - **Editing variant** uses a distinct capsule fill (`NSColor.textBackgroundColor` + 1pt accent stroke, matching macOS Finder inline-rename) and suppresses the `×` close button via `.tabChipShell(..., showsCloseButton: false)`. The shell still reserves the trailing geometry so chip width stays constant when toggling.
    - **Outer `.contextMenu` and the rename VoiceOver rotor action** (via `SelectableTabChip`'s `renameAccessibilityAction:`) are both gated on `editingConversationID == nil` — forbidding rename on *any* tab while another is in-flight. Swapping `editingConversationID` mid-edit left the target chip stuck in editing state without an input field (simultaneous TextField unmount/mount in one SwiftUI update pass didn't converge). The sidebar thread row applies the same guard.
- Conversation tab chips are not list rows, but they mirror the same press-feedback principles: the select action owns the full capsule hit area, trailing affordances (close button) overlay on top, and selected styling uses fill changes rather than capsule strokes (macOS can render stray vertical artifacts from chip outlines in snapshots).
- `ThreadDetailConversationTabs` keeps the system `.bar` background for the header chrome; add any custom separator as an overlay. Replacing the bar with `windowBackgroundColor` creates an unintended dark strip in the live app.

## Keyboard Shortcuts

- **⌘1..⌘9** attach to each chip's visible select `Button` via the optional `KeyboardShortcut?` variant, scoped to the multi-conversation branch (single-conversation headers render a label, no binding).
- **⌘W** does *not* ride on the per-chip close `Button` — per-chip bindings didn't reliably override the system "Close Window" when the first chip was selected. Instead, one invisible `Button` attached as a `.background` to the tab bar carries `⌘W` and calls `onRemove(selectedConversation)`.
    - **Mount unconditionally** (not gated on `conversations.count > 1`) and guard internally: during an inline rename *or* with only one conversation, the action returns early. Closing a thread isn't a concept — ⌘W must absorb the key, not fall through to the scene's default "Close Window" and kill the app window.
    - **Attach via `.background`, not HStack sibling.** A zero-framed HStack child still consumes the HStack's `spacing`, shifting layout and breaking snapshots.
    - **Carry `.id(selectedConversation.persistentModelID)`.** Without selection-scoped identity, SwiftUI keeps the shortcut's bound action captured from the first mount — ⌘W always closes the first tab even after selecting another.
    - **Use an enabled, no-op action rather than `.disabled`** for guarded cases. A disabled shortcut-bearing button lets ⌘W fall through to the scene's default close; an enabled no-op absorbs it.
    - **Keep everything off `.commands` / `CommandGroup`.** Those always surface in the menu bar and can be preempted by the scene's default close handling.
- **Close-tab selection logic**: when `removeConversation` deletes the currently-selected tab, pick the visual neighbor (next, fallback previous) before the delete/save and set via `appState.selectConversation`. `repairSelectedConversationIfNeeded` otherwise falls back through `selectedConversation(in:)`'s main-preference sort, which jumps back to the first tab when closing a non-first tab.
- **`onRemove` re-checks `conversations.count > 1` before setting `pendingDeleteConversation`.** The child ⌘W button's in-closure guard alone is not sufficient — after back-to-back deletions down to a single tab, ⌘W can still trip the confirmation dialog and `removeConversation`'s backend guard surfaces an error banner. The parent-level guard reads fresh state via `ThreadDetailView`'s fetch-backed `conversations` property.

## Async Conversation Removal

- **Capture both `persistentModelID` and the UUID-string `id` synchronously inside the confirmation dialog's `Remove` button closure, and pass both into `removeConversation(id:conversationIDString:)`.** Do not let the async helper re-resolve a `Conversation` via `modelContext.model(for:)` only to read `.id` off it — `model(for:)` can return a zombie reference whose first persisted-property access traps with `_assertionFailure`. See **Async Property Access** in `Alveary/Data/AGENTS.md` for why pre-await capture is safer than re-fetching.

## Scroll Hooks And Sentinel

- **Wrap the multi-conversation `ScrollView` in a `ScrollViewReader` and tag each chip with `.id(conversation.persistentModelID)`.** Mirrors the terminal-pane rule: without an explicit `.id` per chip, `ScrollViewProxy.scrollTo` has no target to register against.
- **The trailing sentinel view (`Color.clear.frame(width: 12)` with `.id(ScrollTarget.trailingSentinel)`) does double duty** — it reserves the 12pt visible breathing gap between the last chip and the overlay divider AND serves as the scroll target whose trailing edge is the content's absolute trailing edge. Required because `proxy.scrollTo(lastChipID)` with default `nil` anchor does minimum-scroll-to-visible — stops when the chip's trailing meets the viewport's trailing, and a separate `.padding(.trailing, 12)` then sits offscreen (last chip visually butts the divider, observed with ⌘T on a full row). With the sentinel *as* the gap, scrolling to it with `anchor: .trailing` lands `contentOffset == tabsMaxScrollableDistance` and the full 12pt stays on-screen.
    - **Structure**: wrap the chip HStack in an outer `HStack(spacing: 0)` with the sentinel as the outer's final element. Do not put the sentinel inside the inner (spacing: 6) HStack — spacing pushes it inward and its trailing edge stops 6pt short of the content trailing edge, reintroducing the clip-at-divider bug.
    - **Do not reintroduce a separate `.padding(.trailing, 12)` on the chip HStack.** The sentinel's 12pt width already reserves the gap; a second padding doubles it to 24pt at scroll-to-end.
- **Scroll on selection via `onChange(of: selectedConversation.persistentModelID, initial: true)`.** If the selected ID matches `conversations.last?.persistentModelID`, target `ScrollTarget.trailingSentinel` with `anchor: .trailing`; otherwise target the chip ID with default `nil` anchor. Mid-row selections use minimum-scroll-to-visible so already-visible chips don't jump; last-chip selections route through the sentinel so the 12pt gap is preserved. `initial: true` covers first render.
- **Also scroll on count-grow via `onChange(of: conversations.count)` guarded on `newCount > oldCount`, targeting the sentinel with `anchor: .trailing`.** `createConversation` appends, so count-grow surfaces the newly-added (last) conversation. Guard skips the removal path. Mirrors the terminal pane's `sessions.count` hook; the conversation row adds the sentinel step because it has a trailing gap the terminal row doesn't.

## Divider Mechanics

- **Only the trailing edge divider exists — no leading divider.** Unlike the terminal pane (fixed terminal icon to the left), the conversation row starts flush against the pane edge, so a leading divider would have no fixed element to abut. The trailing divider sits between the scrolling tabs and the pinned `New Conversation` button and hides once scrolled to the end.
- **Drive the divider off `onScrollGeometryChange`, not `GeometryReader + PreferenceKey`.** `ConversationTabsScrollGeometry` snapshots `contentSize.width`, `containerSize.width`, `contentOffset.x`; `hasTabsBehindTrailingEdge` gates on `effectiveMaxScroll > 0.5 && contentOffset < effectiveMaxScroll - 0.5`, where `effectiveMaxScroll = tabsMaxScrollableDistance - tabsTrailingSentinelWidth`. The subtraction is load-bearing: the sentinel adds 12pt to `contentWidth` but isn't a chip, so the last 12pt of scroll is reserved trailing space. Without it the divider keeps rendering while only the sentinel scrolled into view (no chip clipped), and draws a false-positive when chips nearly fit the viewport and only the sentinel overflows. Share the same tint (`Color.primary.opacity(colorScheme == .dark ? 0.35 : 0.3)`) and 18pt height as the terminal pane.
- **Render as `.overlay(alignment: .trailing)` on the ScrollView.** Overlays don't affect ScrollView layout, so the divider appears/disappears without shifting tab content by 1pt. An inline HStack sibling moves the divider out of the ScrollView's frame — the space left of the divider is supposed to live *inside* the scrollable content.
    - **Do not use `.contentMargins(.trailing, N, for: .scrollContent)`.** On macOS 26 it doesn't reserve visible trailing space in this layout (repro: chip fill measured 0.5pt from divider in the overflow snapshot). The trailing sentinel is the working alternative.

## Outer Layout And Regression Guard

- **Keep the ScrollView greedy (`.frame(maxWidth: .infinity)`) in the multi-conversation branch; render `Spacer()` only in the single-conversation label branch.** `New Conversation` right-aligns because the ScrollView consumes the remaining width and the button carries `.padding(.leading, 12)`. Do not reintroduce a sibling `Spacer()` alongside a flexible ScrollView — the trailing divider floats mid-row instead of hugging the button.
- **The 20pt pane-edge inset lives *inside* the scrollable content, not on the outer tab-row.** Outer row uses `.padding(.trailing, 20)` only; the multi-conversation HStack carries `.padding(.leading, 20)` so the first chip appears 20pt in at `contentOffset == 0` while letting chips scroll under the pane's visible leading edge. The single-conversation label branch applies `.padding(.leading, 20)` to the label directly.
    - **Why**: chips scroll past the pane's visible leading edge — they clip against the pane boundary, not a 20pt inner boundary. Moving the inset inward preserves the 20pt visual at `contentOffset == 0`.
- **The 12pt left of the divider is the trailing sentinel's width, not a reserved scroll region.** At scroll-to-end, a 12pt gap sits between the last chip and the overlay divider. Mid-scroll, any chip being clipped by the trailing edge sits flush against the divider — that clip-flush behavior is intentional: the gap lives inside the scrollable content and only shows when the content's natural trailing reaches the visible area. Pair with `.padding(.leading, 12)` on `New Conversation` for 12pt right of the divider (outside the scrollable content).
    - **Why**: the user wanted the space before the divider to be inside the scrollable content rather than reserved in a non-scrollable band. A sibling divider or `contentMargins`-reserved band would produce a permanent gap at mid-scroll that the user rejected.
- **`testConversationTabsOverflow` is the regression guard for the greedy-ScrollView layout, not the divider.** 8 tabs at 500pt width force the overflow state, pinning the flexible ScrollView + trailing `New Conversation` layout. Reintroducing a sibling `Spacer()` would shift the button and fail verification.
    - **The trailing-edge divider is NOT in this baseline.** `onScrollGeometryChange` dispatches async on a later runloop tick, after the snapshot pass's `displayIfNeeded()`, so `hasTabsBehindTrailingEdge` is still `false` at capture time. Regressions to `overlay(alignment: .trailing)` / `onScrollGeometryChange` / `hasTabsBehindTrailingEdge` will **not** be caught by this test — verify manually in the running app. The terminal pane's `testTerminalPaneSessionsOverflow` *does* capture its equivalent divider (different layout timing stabilizes geometry before the displayIfNeeded boundary); don't assume parity.
