## Shared View Components

These rules cover shared UI components under `Alveary/Views/Components/`.

## Selectable Row Gestures

- `SelectableRowModifier` in `SelectionRowBackground.swift` drives both the press highlight and the click action from a single `DragGesture(minimumDistance: 0)`: `onChanged` sets the highlight, `onEnded` clears it and calls the row action as long as the pointer didn't move far. Do not revert to `.onTapGesture` (alone or alongside `.onLongPressGesture(minimumDuration: .infinity, pressing:)`): SwiftUI's tap gesture on macOS stops firing when a click is held past its short-click window, so the background highlights on mouse-down but mouse-up after a longer hold does nothing. Keep a sibling `.accessibilityAction { action() }` so VoiceOver activation still invokes the row.

## TabChipButtonStyle

- Use `TabChipButtonStyle(isSelected:)` from `TabChipButtonStyle.swift` for any pill/capsule-shaped select affordance (conversation tab chips, terminal session chips, etc.) so the whole capsule is the hit area — not just the text. Pair it with a trailing `ZStack`-overlaid close button (with `focusEffectDisabled()`) and outer padding on the button's content rather than the chip's HStack, so the capsule's pressed fill covers the full visual bounds. Do not revert to wrapping the label in a `.plain` `Button` inside a separate `HStack` with a capsule `.background`; that collapses the hit area to the text and leaves the rest of the capsule dead.

## SelectableTabChip

- `SelectableTabChip` (`SelectableTabChip.swift`) is the shared shell that `ConversationTabChip.selectableChip` and `TerminalSessionChip` both render. It owns the status-dot + markdown label + close-button capsule and the `TabChipButtonStyle(isSelected:)` wiring, so the two surfaces cannot drift on padding, accessibility traits, or chip-color rules. Status-dot color semantics (blue = in-progress, green = done, etc.) are owned by the **Status Dot Colors** section in `Alveary/Views/AGENTS.md`.
- **Always build new tab/chip surfaces on top of `SelectableTabChip`** rather than hand-rolling another ZStack with a `TabChipButtonStyle` button. If a new surface needs extra affordances (e.g. inline rename, context menus), layer them around the shared chip via ordinary SwiftUI modifiers — don't fork the structural code. Rename-style custom accessibility actions must route through the built-in `renameAccessibilityAction:` parameter so the action binds to the select button's accessibility element; attaching `.accessibilityAction(named:)` on the wrapping view binds to the outer container instead and hides the action from users traversing elements individually.
- Surfaces that share the outer geometry but render a different inner content (`ConversationTabChip.editingChip` renders a `TextField`, not a `Button`) should reuse the `.tabChipContentLayout()` and `.tabChipShell(closeAccessibilityLabel:closeHelpText:onClose:showsCloseButton:)` View modifiers exposed from the same file rather than duplicating the padding/ZStack/fixedSize scaffolding. Pass `showsCloseButton: false` to suppress the `×` while preserving the 36pt trailing dead-space, so switching into that mode does not resize the chip.
- The trailing `×` is rendered by the private `TabChipCloseButton` view in `SelectableTabChip.swift`, which owns its own `@State` hover flag — the icon lightens and a subtle circular background fades in on hover, matching the `SidebarProjectsHeaderRow` +-button treatment. Pass a non-nil `closeHelpText` (routed via `SelectableTabChip.closeHelpText` or the shell modifier's matching parameter) to render a tooltip; the conversation-tab shell uses `"Close Conversation (⌘W)"`, terminal chips leave it nil until a shortcut is assigned. Nil suppresses `.help` entirely via `OptionalHelp` so macOS doesn't render an empty-tooltip popover.
- The chip renders its label through `AppMarkdownInlineLabel`, whose inline-code chip fill is always `.standard` (uniform across selection). This is the carve-out documented under **Accent Color Surfaces** below — surfaces that want selection-aware chip swapping should use `AppMarkdownInlineCodeChip(..., style: ...)` directly rather than `SelectableTabChip`.

## AppMarkdownInlineLabel

- Use `AppMarkdownInlineLabel` (not raw `Text`) for any single-line label that renders a user-provided string which may contain inline markdown code or `@file` mentions, such as sidebar thread rows and conversation tab chips. The label segments the string and renders both inline code and mention tokens via `AppMarkdownInlineCodeChip` clamped to the surrounding text's line height so each chip's rounded background visually overflows into the parent's vertical padding without inflating row/tab height.
- **Mention detection is load-bearing across surfaces.** The label reuses `ChatInputFieldTextSupport.fileMentionMatches(in:)` plus `CanonicalPath.decodeStoredMentionPath(_:)` so thread row and tab titles render a mention with the same `@<basename>` chip shape that the composer and user bubbles use. Mentions that land inside a fenced block or inline-code span stay as code (do not re-chip as a mention) — the filter against `blockRanges + inlineFullRanges` in `segments(for:)` is the invariant that keeps `` `@not-a-file` `` from getting double-stylized.
- The label's `textStyle` parameter drives both the SwiftUI text font and the chip font size; keep them paired by always passing a single `NSFont.TextStyle` rather than reintroducing independent `font` and chip-size parameters.
- When a chip-bearing surface needs an explicit `accessibilityLabel` (for example a `Button`-wrapped chip where SwiftUI's combined children label doesn't apply), use `AppMarkdownInlineLabel.plainText(from:)` — it strips backtick delimiters and decodes mentions to their `@<basename>` display form. Do not read the raw markdown aloud via VoiceOver, and do not hand-roll a regex replacement that can cut inside multi-backtick spans or leave percent-encoded mention paths for VoiceOver to spell out character by character.

## Accent Color Surfaces

- The app's accent color is defined in `Alveary/Assets.xcassets/AccentColor.colorset` and wired to the target via `ASSETCATALOG_COMPILER_GLOBAL_ACCENT_COLOR_NAME: AccentColor` in `project.yml`. To change the accent app-wide, edit the `AccentColor` color set; do not introduce a separate tint or apply `.tint(...)` at the root — SwiftUI's `Color.accentColor` static does not follow environment tint, so only the asset-catalog approach updates every call site consistently.
- Accent-token plumbing: `AppAccentFill.primary`/`.pressed` (in `AppAccentFill.swift`) and `AppMarkdownCodeBlockPalette.composerChipFillNSColor` derive from `NSColor.controlAccentColor` and are the canonical accent surfaces — every prominent amber UI in the app (primary action buttons via `primaryActionButtonStyle()`, user chat bubbles, selected sidebar rows, selected conversation tabs, selected terminal session chips, prompt-block question-card header pills, accent-toned diff preview rows, the scroll-to-latest button, composer inline-code / slash / `@mention` chips, queued-message chips, the "Selected" badge) shares this one token so the app speaks with one accent voice. All of these resolve to the asset-catalog `AccentColor` when the user's macOS accent preference is "Multicolor" (the default); under a specific system accent they track that system color. If you ever need the asset color regardless of the system preference, use `NSColor(named: "AccentColor")`. The `.standard` and `.userBubble` inline-code chip palettes are intentionally *not* accent-derived (see the inline-code bullet below). Raw `Color.accentColor` is reserved for low-level AppKit integration points (e.g. syntax highlighting in `AppTextEditor+CodeBlocks.swift`, drop-target border flashes) where a fully-saturated system accent is semantically appropriate.
- Pair `AppAccentFill.primary` / `.pressed` with `.primary` / `NSColor.labelColor` foreground. `.primary` adapts to both schemes and stays legible against the muted light-mode blend toward white and the dark-mode blend toward black. Destructive buttons are the exception — they fill with a non-accent red and keep `.white` as the foreground.
- Inline-code chips have three `AppMarkdownInlineCodeStyle` variants. `.standard` is the default; `.userBubble` and `.composer` are opt-in for specific parents. An earlier accent-derived default tinted every inline chip amber, which was too noisy outside live-typing UI — keep grayscale everywhere except `.composer`.
    - **`.standard`** — neutral app chrome (unselected sidebar rows, unselected conversation tabs, assistant bubbles). Mid-gray against near-white (light) / mid-gray against dark chrome, foreground `.labelColor`. Tuned for low chip-density surfaces; the composer uses `.composer` because it's the densest.
    - **`.composer`** — accent-backed fill (`composerChipFillNSColor` / `composerChipForegroundNSColor`) for inline-code / slash / `@mention` chips. Reinforces the composer as the live-typing surface. Referenced directly from the NSTextView drawing path in `AppTextEditor+*.swift`. Queue items use `AppMarkdownText(inlineCodeStyle: .composer)` — queue items are composer chrome, not historical gray.
    - **`.userBubble`** — chips on accent-tinted parents (user bubble). Near-white gray (light) / mid-dark gray (dark), foreground `.labelColor`. Do not reintroduce a label-color-at-low-opacity fill: fine over dark accents, disappears into bright ones (amber default). Selected sidebar rows / conversation tabs / terminal chips *do not* use `.userBubble` — they lock to `.standard` so chip color is uniform across selection.
    - **`AppMarkdownInlineLabel` always renders `.standard`** — no selection flag. Every single-line chip surface keeps chip color uniform across selection. A future selection-aware surface should use `AppMarkdownInlineCodeChip(..., style: ...)` directly rather than resurrecting the flag.
    - **Tune swatches in `AppMarkdown+Palette.swift`.** `.standard` (~0.12 dark / ~1.0 light) and `.userBubble` use different gray values so each contrasts against its own parent; merging would regress whichever surface it isn't tuned to. `.composer` splits fill/foreground: `composerChipFillNSColor` is aliased to `AppAccentFill.primaryNSColor` (tracks the system accent); `composerChipForegroundNSColor` is `.labelColor` — the same pairing every `AppAccentFill.primary` surface uses. An earlier accent-derived foreground stacked amber-on-amber and read as "all yellow". Retune by adjusting `AppAccentFill`'s blend math, not by introducing a fixed swatch; do not reintroduce an accent-derived foreground.
- Multi-line bubble surfaces (assistant and user chat bubbles) render inline code and composer chips (`/command`, `@file`) through Textual's native inline-code styling — `AttributedString.inlinePresentationIntent = .code` plus the bubble's `InlineStyle.code(.backgroundColor(...))` — **not** `AppMarkdownInlineCodeChip`. Load-bearing for uniform line heights: an attachment-rendered rounded pill grows the line it sits on because SwiftUI `Text` uses the attachment placeholder's height as the run's ascent contribution. Single-line surfaces (sidebar rows, tabs, terminal chips) keep the rounded pill via `AppMarkdownInlineCodeChip` because their overflow absorbs into row padding. Do not reintroduce attachment-rendered inline code in `AppMarkdownParser.attributedString(for:)` without reproducing the uniform-line-height behavior some other way.
- `AccentColor` ships explicit `Any Appearance` and `Dark` slots in its color set, even though both slots currently hold the same value. Retune per-mode by editing the existing dark slot rather than adding the appearance entry from scratch — keeping both slots populated makes future light/dark divergence a value edit instead of a schema change.
- `AppAccentFill.primary` / `.pressed` are scheme-aware via a dynamic `NSColor` provider: dark mode blends the accent toward black (saturated deeper shade), light mode toward white (muted tint). Both sides are fully opaque — a prior alpha-tint implementation showed ghost content through floating UI like the scroll-to-latest button. Prefer `AppAccentFill.primary` over fresh `Color.accentColor.opacity(...)`; a naive bright-accent fill reads as over-saturated (light) or washes out (dark). The composer chip fill (`AppMarkdownCodeBlockPalette.composerChipFillNSColor`) is a structural alias of `AppAccentFill.primaryNSColor` so prominent chrome and live-typing chrome can't drift apart — if they ever should diverge, replace the alias with a separate definition rather than duplicating the blend math.
- SwiftUI `Link` does *not* pick up `Color.accentColor` automatically — it renders in system link blue regardless of the app's `AccentColor` asset. Apply `.foregroundStyle(Color.accentColor)` explicitly on every `Link` so it matches the rest of the accent surfaces (e.g. the GitHub-repo link in `ProjectSettingsView+RepositoryCard`).

## Textual Style Modifier Ordering

- Do not apply `.textual.structuredTextStyle(.default)` alongside `.textual.inlineStyle(...)` or `.textual.codeBlockStyle(...)` on the same view. `structuredTextStyle(.default)` writes every per-style environment key (including `\.inlineStyle` and `\.codeBlockStyle`) to the library defaults, and in a SwiftUI modifier chain the env modifier closest to the target view wins — so the default overwrites the custom inline/code-block style that comes after it in the chain. Symptom: blue Markdown links and default (not `AppMarkdownCodeBlockStyle`) code-block rendering even though `appMarkdownInlineStyle` and the custom style are in the chain.
- Fix by dropping the `.structuredTextStyle(.default)` call. The per-style environment keys already default to `.default`, so removing it only unblocks the downstream overrides. `AppMarkdownText` in `AppMarkdown.swift` has a comment noting this.

## AppMarkdownCodeBlockPalette

- The palette exposes scheme-aware colors as cached `static let NSColor` values (e.g. `AppMarkdownCodeBlockPalette.inlineFillNSColor`). The underlying `NSColor(name:dynamicProvider:)` resolves per-appearance at draw time; do not reintroduce a `(for: ColorScheme)` parameter. Caching as `static let` is load-bearing for tests that round-trip colors through `NSAttributedString` attributes — two dynamic `NSColor` instances are not `==`, so a fresh construction each access would fail `XCTAssertEqual`.

## Accent-Derived Dynamic Colors

- `NSColor.accentDerived(transform:)` in `NSColor+AccentDerived.swift` is the canonical way to build a dynamic `NSColor` whose value depends on the resolved accent. The helper resolves `NSColor.controlAccentColor` to a concrete sRGB color *inside* `performAsCurrentDrawingAppearance`, then hands that flattened accent to `transform` so operations like `withAlphaComponent(_:)` / `blended(withFraction:of:)` compute against the requested appearance rather than whichever appearance happens to be current (popovers, off-screen snapshots, test hosts). Do not roll your own `NSColor(name:dynamicProvider:)` that reads `controlAccentColor` and then operates on it outside the block — capturing the dynamic reference without flattening makes the subsequent blend/alpha math race with whatever appearance AppKit picks when the result is rendered.
- The light/dark branch lives inside the `transform` closure and switches on `appearance.bestMatch(from: [.darkAqua, .aqua])`. `AppAccentFill.primary` / `.pressed` and `AppMarkdownCodeBlockPalette.composerChip*` all follow this pattern; copy it rather than reintroducing a helper that takes a `ColorScheme` argument. The `.standard` and `.userBubble` inline-code chip swatches do *not* use `.accentDerived` — they're pure grayscale built via `NSColor(name:dynamicProvider:)`, which is the right helper when the result does not depend on the resolved accent.
- When a dynamic `NSColor` needs to feed Textual's `DynamicColor`, use `NSColor.resolved(for:)` (in `NSColor+AccentDerived.swift`) against `.aqua` and `.darkAqua` and pass the flattened `Color`s to `DynamicColor(light:dark:)`. `AppMarkdown.swift`'s `dynamicColor(from:)` helper already does this for the inline-code styles. Avoid `DynamicColor(Color(nsColor: dynamic))` — it works today because SwiftUI preserves the NSColor's `dynamicProvider` through `Color(nsColor:)`, but that's an implicit bridging contract; flattening to explicit light/dark variants keeps the scheme switch inside `DynamicColor`'s own branch. Consequence: Textual-rendered chips won't reflect a live macOS system-accent change until the app relaunches, which is acceptable since the asset-catalog `AccentColor` (default "Multicolor") is static per launch. The direct `AppMarkdownInlineCodeChip` view renders through `Color(nsColor:)` at view-body time and does not need flattening.

## AppTextEditor And AppKitTextView

- Keep the placeholder drawn inside `AppKitTextView` instead of reintroducing a SwiftUI overlay so it shares the real `NSTextView` insets and caret positioning.
- `AppKitTextView.allowsVibrancy` is hard-coded `false`. The composer hosts this text view inside a SwiftUI card whose background is `.fill(.bar)` — an `NSVisualEffectView`-backed material — and `NSTextView` opts into vibrancy by default, so the inline-code / slash / `@mention` chip fills drawn via `setFill()` / `NSBezierPath.fill()` in `draw(_:)` would composite against that backdrop and shift the apparent color away from the literal `NSColor`. User bubbles, primary buttons, selected rows, and selected tab chips draw `AppAccentFill.primary` through pure SwiftUI shapes with no vibrancy participation — the override is what keeps the composer chip fill visually aligned with every other `AppAccentFill.primary` surface. Do not make the override conditional on "is the parent a material?"; editors not embedded in a material backdrop receive no vibrancy in the first place, so disabling it there is a no-op and the simpler unconditional override is safer against future surfaces (sheets, popovers) that introduce a material parent.
- Selection-change callbacks for `AppKitTextView` must not synchronously trigger layout-dependent restyling such as chip rect calculation or `NSLayoutManager.ensureLayout(for:)`; update lightweight typing state inline, but defer full restyles to the next main-runloop turn to avoid AppKit reentrancy traps.
- The `focus: FocusState<Bool>.Binding?` parameter on `AppTextEditor` is a one-way AppKit→SwiftUI bridge: `handleFocusChange` backfills the binding after `becomeFirstResponder`/`resignFirstResponder`. Programmatic focus claims must route through the `requestFirstResponder: UUID?` + `onFocusRequestConsumed` token plumbing, not through writes to the `@FocusState` binding. See the **Focus And Keyboard Coordination** section in `Alveary/Views/AGENTS.md` for the full rationale and cross-surface pattern.
- Features that need to READ "is the composer first responder?" during a SwiftUI body evaluation (for example `inlineSlashCommandHint` in `ChatInputField`) must drive off the plain `isAppKitFirstResponder: Binding<Bool>?` parameter, not the `@FocusState` binding. Writes to a `@FocusState` binding without a matching `.focused($state)` anchor in the view hierarchy update the underlying storage but do not reliably invalidate view descendants, so a computed property reading `isInputFocused` after an AppKit-side write can still observe a stale `false`. The coordinator's `handleFocusChange` mirrors every state transition into both the `@FocusState` (for publishing via `focusedSceneValue(\.chatComposerFocus, ...)`) and the plain binding (for local reads). Keep both in sync — do not drop one on the assumption the other is enough.
- Trailing kern for chips is asymmetric on purpose: `applyTrailingKern` is only called for `.slashCommand` chips in `applyTextChips`. Slash commands anchor to the line's leading edge, so their trailing edge is where the caret, next typed character, and inline hint sit — giving them `.kern: appTextEditorChipTrailingKern` adds the visual breathing room and keeps those three things aligned. File mentions and inline code sit mid-line between word-separating spaces, so adding trailing kern there would produce asymmetric "room after, none before" spacing. The chip's rounded background automatically widens via `enumerateEnclosingRects` including the kerned advance — that's the intended look (chip has visible trailing padding, caret/hint/text at chip trailing edge), matching how inline code chips read in the composer. Note: `.fileMention` chips in *compact* mode do apply a different `.kern` (negative, per-char) to shrink the stored-glyph width to the decoded label's width — see the compact-chip bullet below. The two kern mechanisms don't collide because `applyTrailingKern` is gated on `style == .slashCommand`.
- `claimFirstResponder(on:retriesRemaining:)` polls a handful of main-runloop ticks because `updateNSView` can run before the `NSTextView` is inside an `NSWindow` (a brand-new composer mount can interleave the window attachment across render passes). A synchronous `window?.makeFirstResponder` on a window-less view silently no-ops; the retry turns the race into a wait. Do not replace the retry with a longer one-shot `Task.sleep` — the correct wait depends on how fast the hierarchy gets composed, and the retry short-circuits as soon as `textView.window` becomes available.
- The retry chain is deduplicated via `firstResponderClaimInFlight` on the coordinator: `claimFirstResponder` returns immediately if a prior chain is still running, and the flag only clears on a terminal outcome (already first responder, successful `makeFirstResponder`, or retries exhausted). Without this, every `updateNSView` pass that ran while the window was still unattached would schedule a parallel retry chain — harmless but wasteful. Keep the flag's writes inside the `DispatchQueue.main.async` body so they stay on the main actor, and do not clear it between recursive `scheduleFirstResponderClaim` hops.
- `AppKitTextView.updateDragTypeRegistration()` is overridden so a parent SwiftUI `.dropDestination(for: URL.self)` can receive drops instead of the text view. The behavior is **opt-in per editor** via the `disablesAppKitDragDestination: Bool = false` parameter on `AppTextEditor` / `AppKitTextEditorView`.
    - **Only set `disablesAppKitDragDestination: true` when a parent view has a `.dropDestination` to catch drops.** The composer (`ChatInputField`) opts in because it owns an outer `.dropDestination(for: URL.self)` that runs `handleDroppedFiles` to insert `@`-prefixed mention chips. Editors without a parent drop target (Skills instruction editor, MCP headers/env) must keep the default `false` — otherwise drops into them silently disappear.
    - **Override the hook, don't just unregister once.** `NSTextView` calls `updateDragTypeRegistration()` whenever relevant state changes (`isRichText`, `importsGraphics`, etc.), which re-registers every `readablePasteboardType`. A one-shot `unregisterDraggedTypes()` in `makeNSView` gets silently reverted on the next state change — the override is what makes the unregister sticky.
    - **Don't keep `.string` registered either when opting in.** Finder drags carry the dropped path as `.string` alongside `.fileURL`, so re-registering `.string` still lets the text view swallow the drop and insert the raw path inline. Unregister everything; `.dropDestination(for: URL.self)` handles the file drop and paste (⌘V) still flows through `readablePasteboardTypes`.
    - **Don't override `readablePasteboardTypes`.** That key also gates paste; returning an empty list breaks ⌘V too.
- Compact file-mention chip rendering in `applyCompactFileMentionAttributes` / `AppKitTextView.drawCompactChipLabels` collapses a percent-encoded stored path down to a decoded `@<basename>` label:
    - **Hide the stored glyphs with `.foregroundColor: NSColor.clear`** over the entire chip range — the decoded label is drawn separately, so no part of the stored text should render. Don't revert to the older "hide prefix, show stored suffix" trick: stored text is now percent-encoded (see `Alveary/Views/Input/AGENTS.md`), so its tail no longer matches the decoded filename and slicing the stored text produces garbage like `@shot%202026...`.
    - **Shrink the chip rect with a computed negative `.kern`.** Per-char reduction = `charAdvance * (storedLength - decodedLength) / storedLength`, applied to the whole chip range. `NSLayoutManager.enumerateEnclosingRects` reflects `.kern` in its enclosing rects, so both the chip background rect and the label draw rect collapse to the decoded width. `.expansion` was tried first — it scales glyphs visually but doesn't show up in `enumerateEnclosingRects`, leaving the chip background stretched past the label.
    - **Draw the decoded label in `AppKitTextView.drawCompactChipLabels`.** The method runs after `super.draw(_:)` so the (clear) stored glyphs have already rendered; it then paints `CanonicalPath.decodeStoredMentionPath(chip.displayText)` over the chip's enclosing rect. Compact mode is gated on a single-line chip (`rects.count == 1` in `textChipDisplayMode`), so there's at most one rect per chip — don't call the label drawer across multiple line fragments.
