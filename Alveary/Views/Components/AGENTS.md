## Shared View Components

These rules cover shared UI components under `Alveary/Views/Components/`.

## Selectable Row Gestures

- `SelectableRowModifier` in `SelectionRowBackground.swift` drives both the press highlight and the click action from a single `DragGesture(minimumDistance: 0)`: `onChanged` sets the highlight, `onEnded` clears it and calls the row action as long as the pointer didn't move far. Do not revert to `.onTapGesture` (alone or alongside `.onLongPressGesture(minimumDuration: .infinity, pressing:)`): SwiftUI's tap gesture on macOS stops firing when a click is held past its short-click window, so the background highlights on mouse-down but mouse-up after a longer hold does nothing. Keep a sibling `.accessibilityAction { action() }` so VoiceOver activation still invokes the row.

## TabChipButtonStyle

- Use `TabChipButtonStyle(isSelected:)` from `TabChipButtonStyle.swift` for any pill/capsule-shaped select affordance (conversation tab chips, terminal session chips, etc.) so the whole capsule is the hit area â€” not just the text. Pair it with a trailing `ZStack`-overlaid close button (with `focusEffectDisabled()`) and outer padding on the button's content rather than the chip's HStack, so the capsule's pressed fill covers the full visual bounds. Do not revert to wrapping the label in a `.plain` `Button` inside a separate `HStack` with a capsule `.background`; that collapses the hit area to the text and leaves the rest of the capsule dead.

## AppMarkdownInlineLabel

- Use `AppMarkdownInlineLabel` (not raw `Text`) for any single-line label that renders a user-provided string which may contain inline markdown code, such as sidebar thread rows and conversation tab chips. The label segments the string and renders inline code via `AppMarkdownInlineCodeChip` clamped to the surrounding text's line height so the chip's rounded background visually overflows into the parent's vertical padding without inflating row/tab height.
- The label's `textStyle` parameter drives both the SwiftUI text font and the chip font size; keep them paired by always passing a single `NSFont.TextStyle` rather than reintroducing independent `font` and chip-size parameters.
- When a chip-bearing surface needs an explicit `accessibilityLabel` (for example a `Button`-wrapped chip where SwiftUI's combined children label doesn't apply), use `AppMarkdownInlineLabel.plainText(from:)` to strip backtick delimiters. Do not read the raw markdown aloud via VoiceOver, and do not hand-roll a regex replacement that can cut inside multi-backtick spans.

## Accent Color Surfaces

- The app's accent color is defined in `Alveary/Assets.xcassets/AccentColor.colorset` and wired to the target via `ASSETCATALOG_COMPILER_GLOBAL_ACCENT_COLOR_NAME: AccentColor` in `project.yml`. To change the accent app-wide, edit the `AccentColor` color set; do not introduce a separate tint or apply `.tint(...)` at the root â€” SwiftUI's `Color.accentColor` static does not follow environment tint, so only the asset-catalog approach updates every call site consistently.
- Accent-token plumbing: SwiftUI `Color.accentColor` is used for solid fills (primary buttons, the scroll-to-latest pill), while `AppSelectionStyle.rowFill`/`pressedFill` and `AppMarkdownCodeBlockPalette`'s chip colors derive from `NSColor.controlAccentColor`. Both resolve to the asset-catalog `AccentColor` when the user's macOS accent preference is "Multicolor" (the default). If the user picks a specific system accent in System Settings, both APIs track that system color, so they stay in sync â€” but neither follows the asset literally in that case. If you ever need the asset color regardless of the system preference, use `NSColor(named: "AccentColor")`.
- For any *solid*-accent filled surface (e.g. primary action buttons, the scroll-to-latest pill) with text or an icon on top, use `Color.onAccent` (defined in `Color+OnAccent.swift`) as the foreground instead of hard-coded `.white`. `onAccent` is a dynamic `NSColor` that resolves `NSColor.controlAccentColor` at draw time and picks near-black or white by luminance, so it stays legible whether the asset-catalog `AccentColor` is in use (default "Multicolor" preference) or macOS is tracking a dark system accent like graphite or blue. Do *not* replace `onAccent` with a fixed asset or a hard-coded `.white` â€” a fixed-dark asset fails against dark system accents, and a fixed `.white` fails against a bright asset-catalog accent. Destructive buttons are the exception â€” they fill with a non-accent red, so they keep `.white` as the foreground.
- For filled-but-muted accent surfaces (user chat bubble, selected sidebar rows, selected conversation tabs, selected terminal session chips, the placeholder "Selected" badge), reuse `AppSelectionStyle.rowFill` from `SelectionRowBackground.swift` rather than introducing a new opacity-tinted accent fill. Keeping these surfaces on the same token ensures they stay visually coherent as the accent changes.
- Pair foreground tokens to the fill kind: `.onAccent` goes on solid `Color.accentColor` fills; `.primary` (or `NSColor.labelColor`) goes on `AppSelectionStyle.rowFill` / `pressedFill`. `.primary` adapts to both schemes and keeps legibility on the muted dark-mode tint and the darker light-mode fill â€” `.onAccent` is tuned against the raw accent and would clash with the muted variants.
- Inline code chips default to the `.standard` `AppMarkdownInlineCodeStyle`. `AppMarkdownCodeBlockPalette` draws the `.standard` fill from `NSColor.controlAccentColor`: dark mode blends the accent with black (a darker shade, not just a lower-alpha tint) then softens with a slight alpha; light mode uses a direct accent tint at reduced alpha. Foreground is solid accent in dark mode and accent blended toward black in light mode. When a chip sits on top of an accent-tinted parent (user bubble, selected sidebar row, selected conversation tab, selected terminal session chip), switch to the `.userBubble` style â€” a grayscale fill (near-white in light mode, mid-dark gray in dark mode) with `NSColor.labelColor` as the foreground so the chip contrasts sharply with the accent-tinted parent. Do not reintroduce a label-color-at-low-opacity fill for `.userBubble`: it looks fine over darker accents but disappears into bright accent surfaces (e.g. the default amber) because the parent is already saturated. `AppMarkdownInlineLabel(text:isSelected:)` takes a Boolean that picks the right style for row/tab rendering; thread the row/tab's selection state through rather than hard-coding `.standard` or `.userBubble` at the call site. The *exact* opacity and blend values are in `AppMarkdown+Palette.swift` â€” tune there.
- Multi-line bubble surfaces (assistant and user chat bubbles) render inline code and composer chips (`/command`, `@file`) through Textual's native inline-code styling â€” a flat monospaced highlight driven by `AttributedString.inlinePresentationIntent = .code` plus the bubble's `InlineStyle.code(.backgroundColor(...))`. They do *not* go through `AppMarkdownInlineCodeChip`. This is load-bearing for uniform line heights: an attachment-rendered rounded pill grows the line it sits on because SwiftUI `Text` uses the attachment placeholder's height as the run's ascent contribution, and `Canvas.draw(symbol, in:)` scales the chip to the placeholder rect (so shrinking the placeholder compresses the chip). Single-line surfaces (sidebar rows, tabs, terminal chips) keep the rounded pill via `AppMarkdownInlineCodeChip` because their overflow absorbs into row padding. Do not reintroduce attachment-rendered inline code in `AppMarkdownParser.attributedString(for:)` without reproducing the uniform-line-height behavior some other way.
- `AccentColor` ships explicit `Any Appearance` and `Dark` slots in its color set, even though both slots currently hold the same value. Retune per-mode by editing the existing dark slot rather than adding the appearance entry from scratch â€” keeping both slots populated makes future light/dark divergence a value edit instead of a schema change. `OnAccent` is *not* an asset; it's code-derived from the resolved accent luminance (see bullet above), so there is nothing to retune in the asset catalog for it.
- `AppSelectionStyle.rowFill` and `pressedFill` are scheme-aware via a dynamic `NSColor` provider: in dark mode they tint the window background with low-opacity accent; in light mode they render a solid darkened accent so `.primary` text on selected rows, selected conversation tabs, and user chat bubbles retains contrast. When introducing a new selection surface, prefer these tokens over a fresh `Color.accentColor.opacity(...)` â€” an opacity-tinted bright accent over a light background produces a pale fill that fails contrast with foreground text.
- SwiftUI `Link` does *not* pick up `Color.accentColor` automatically â€” it renders in system link blue regardless of the app's `AccentColor` asset. Apply `.foregroundStyle(Color.accentColor)` explicitly on every `Link` so it matches the rest of the accent surfaces (e.g. the GitHub-repo link in `ProjectSettingsView+RepositoryCard`).

## Textual Style Modifier Ordering

- Do not apply `.textual.structuredTextStyle(.default)` alongside `.textual.inlineStyle(...)` or `.textual.codeBlockStyle(...)` on the same view. `structuredTextStyle(.default)` writes every per-style environment key (including `\.inlineStyle` and `\.codeBlockStyle`) to the library defaults, and in a SwiftUI modifier chain the env modifier closest to the target view wins â€” so the default overwrites the custom inline/code-block style that comes after it in the chain. Symptom: blue Markdown links and default (not `AppMarkdownCodeBlockStyle`) code-block rendering even though `appMarkdownInlineStyle` and the custom style are in the chain.
- Fix by dropping the `.structuredTextStyle(.default)` call. The per-style environment keys already default to `.default`, so removing it only unblocks the downstream overrides. `AppMarkdownText` in `AppMarkdown.swift` has a comment noting this.

## AppMarkdownCodeBlockPalette

- The palette exposes scheme-aware colors as cached `static let NSColor` values (e.g. `AppMarkdownCodeBlockPalette.inlineForegroundNSColor`). The underlying `NSColor(name:dynamicProvider:)` resolves per-appearance at draw time; do not reintroduce a `(for: ColorScheme)` parameter. Caching as `static let` is load-bearing for tests that round-trip colors through `NSAttributedString` attributes â€” two dynamic `NSColor` instances are not `==`, so a fresh construction each access would fail `XCTAssertEqual`.

## Accent-Derived Dynamic Colors

- `NSColor.accentDerived(transform:)` in `Color+OnAccent.swift` is the canonical way to build a dynamic `NSColor` whose value depends on the resolved accent. The helper resolves `NSColor.controlAccentColor` to a concrete sRGB color *inside* `performAsCurrentDrawingAppearance`, then hands that flattened accent to `transform` so operations like `withAlphaComponent(_:)` / `blended(withFraction:of:)` compute against the requested appearance rather than whichever appearance happens to be current (popovers, off-screen snapshots, test hosts). Do not roll your own `NSColor(name:dynamicProvider:)` that reads `controlAccentColor` and then operates on it outside the block â€” capturing the dynamic reference without flattening makes the subsequent blend/alpha math race with whatever appearance AppKit picks when the result is rendered.
- The light/dark branch lives inside the `transform` closure and switches on `appearance.bestMatch(from: [.darkAqua, .aqua])`. `AppMarkdownCodeBlockPalette.inlineFillNSColor` / `inlineForegroundNSColor` and `AppSelectionStyle.rowFill` / `pressedFill` all follow this pattern; copy it rather than reintroducing a helper that takes a `ColorScheme` argument.
- When a dynamic `NSColor` needs to feed Textual's `DynamicColor`, use `NSColor.resolved(for:)` (in `Color+OnAccent.swift`) against `.aqua` and `.darkAqua` and pass the flattened `Color`s to `DynamicColor(light:dark:)`. `AppMarkdown.swift`'s `dynamicColor(from:)` helper already does this for the inline-code styles. Avoid `DynamicColor(Color(nsColor: dynamic))` â€” it works today because SwiftUI preserves the NSColor's `dynamicProvider` through `Color(nsColor:)`, but that's an implicit bridging contract; flattening to explicit light/dark variants keeps the scheme switch inside `DynamicColor`'s own branch. Consequence: Textual-rendered chips won't reflect a live macOS system-accent change until the app relaunches, which is acceptable since the asset-catalog `AccentColor` (default "Multicolor") is static per launch. The direct `AppMarkdownInlineCodeChip` view renders through `Color(nsColor:)` at view-body time and does not need flattening.

## AppTextEditor And AppKitTextView

- Keep the placeholder drawn inside `AppKitTextView` instead of reintroducing a SwiftUI overlay so it shares the real `NSTextView` insets and caret positioning.
- Selection-change callbacks for `AppKitTextView` must not synchronously trigger layout-dependent restyling such as chip rect calculation or `NSLayoutManager.ensureLayout(for:)`; update lightweight typing state inline, but defer full restyles to the next main-runloop turn to avoid AppKit reentrancy traps.
- The `focus: FocusState<Bool>.Binding?` parameter on `AppTextEditor` is a one-way AppKitâ†’SwiftUI bridge: `handleFocusChange` backfills the binding after `becomeFirstResponder`/`resignFirstResponder`. Programmatic focus claims must route through the `requestFirstResponder: UUID?` + `onFocusRequestConsumed` token plumbing, not through writes to the `@FocusState` binding. See the **Focus And Keyboard Coordination** section in `Alveary/Views/AGENTS.md` for the full rationale and cross-surface pattern.
- Features that need to READ "is the composer first responder?" during a SwiftUI body evaluation (for example `inlineSlashCommandHint` in `ChatInputField`) must drive off the plain `isAppKitFirstResponder: Binding<Bool>?` parameter, not the `@FocusState` binding. Writes to a `@FocusState` binding without a matching `.focused($state)` anchor in the view hierarchy update the underlying storage but do not reliably invalidate view descendants, so a computed property reading `isInputFocused` after an AppKit-side write can still observe a stale `false`. The coordinator's `handleFocusChange` mirrors every state transition into both the `@FocusState` (for publishing via `focusedSceneValue(\.chatComposerFocus, ...)`) and the plain binding (for local reads). Keep both in sync â€” do not drop one on the assumption the other is enough.
- Trailing kern for chips is asymmetric on purpose: `applyTrailingKern` is only called for `.slashCommand` chips in `applyTextChips`. Slash commands anchor to the line's leading edge, so their trailing edge is where the caret, next typed character, and inline hint sit â€” giving them `.kern: appTextEditorChipTrailingKern` adds the visual breathing room and keeps those three things aligned. File mentions and inline code sit mid-line between word-separating spaces, so adding trailing kern there would produce asymmetric "room after, none before" spacing. The chip's rounded background automatically widens via `enumerateEnclosingRects` including the kerned advance â€” that's the intended look (chip has visible trailing padding, caret/hint/text at chip trailing edge), matching how inline code chips read in the composer. Note: `.fileMention` chips in *compact* mode do apply a different `.kern` (negative, per-char) to shrink the stored-glyph width to the decoded label's width â€” see the compact-chip bullet below. The two kern mechanisms don't collide because `applyTrailingKern` is gated on `style == .slashCommand`.
- `claimFirstResponder(on:retriesRemaining:)` polls a handful of main-runloop ticks because `updateNSView` can run before the `NSTextView` is inside an `NSWindow` (a brand-new composer mount can interleave the window attachment across render passes). A synchronous `window?.makeFirstResponder` on a window-less view silently no-ops; the retry turns the race into a wait. Do not replace the retry with a longer one-shot `Task.sleep` â€” the correct wait depends on how fast the hierarchy gets composed, and the retry short-circuits as soon as `textView.window` becomes available.
- The retry chain is deduplicated via `firstResponderClaimInFlight` on the coordinator: `claimFirstResponder` returns immediately if a prior chain is still running, and the flag only clears on a terminal outcome (already first responder, successful `makeFirstResponder`, or retries exhausted). Without this, every `updateNSView` pass that ran while the window was still unattached would schedule a parallel retry chain â€” harmless but wasteful. Keep the flag's writes inside the `DispatchQueue.main.async` body so they stay on the main actor, and do not clear it between recursive `scheduleFirstResponderClaim` hops.
- `AppKitTextView.updateDragTypeRegistration()` is overridden so a parent SwiftUI `.dropDestination(for: URL.self)` can receive drops instead of the text view. The behavior is **opt-in per editor** via the `disablesAppKitDragDestination: Bool = false` parameter on `AppTextEditor` / `AppKitTextEditorView`.
    - **Only set `disablesAppKitDragDestination: true` when a parent view has a `.dropDestination` to catch drops.** The composer (`ChatInputField`) opts in because it owns an outer `.dropDestination(for: URL.self)` that runs `handleDroppedFiles` to insert `@`-prefixed mention chips. Editors without a parent drop target (Skills instruction editor, MCP headers/env) must keep the default `false` â€” otherwise drops into them silently disappear.
    - **Override the hook, don't just unregister once.** `NSTextView` calls `updateDragTypeRegistration()` whenever relevant state changes (`isRichText`, `importsGraphics`, etc.), which re-registers every `readablePasteboardType`. A one-shot `unregisterDraggedTypes()` in `makeNSView` gets silently reverted on the next state change â€” the override is what makes the unregister sticky.
    - **Don't keep `.string` registered either when opting in.** Finder drags carry the dropped path as `.string` alongside `.fileURL`, so re-registering `.string` still lets the text view swallow the drop and insert the raw path inline. Unregister everything; `.dropDestination(for: URL.self)` handles the file drop and paste (âŒ˜V) still flows through `readablePasteboardTypes`.
    - **Don't override `readablePasteboardTypes`.** That key also gates paste; returning an empty list breaks âŒ˜V too.
- Compact file-mention chip rendering in `applyCompactFileMentionAttributes` / `AppKitTextView.drawCompactChipLabels` collapses a percent-encoded stored path down to a decoded `@<basename>` label:
    - **Hide the stored glyphs with `.foregroundColor: NSColor.clear`** over the entire chip range â€” the decoded label is drawn separately, so no part of the stored text should render. Don't revert to the older "hide prefix, show stored suffix" trick: stored text is now percent-encoded (see `Alveary/Views/Input/AGENTS.md`), so its tail no longer matches the decoded filename and slicing the stored text produces garbage like `@shot%202026...`.
    - **Shrink the chip rect with a computed negative `.kern`.** Per-char reduction = `charAdvance * (storedLength - decodedLength) / storedLength`, applied to the whole chip range. `NSLayoutManager.enumerateEnclosingRects` reflects `.kern` in its enclosing rects, so both the chip background rect and the label draw rect collapse to the decoded width. `.expansion` was tried first â€” it scales glyphs visually but doesn't show up in `enumerateEnclosingRects`, leaving the chip background stretched past the label.
    - **Draw the decoded label in `AppKitTextView.drawCompactChipLabels`.** The method runs after `super.draw(_:)` so the (clear) stored glyphs have already rendered; it then paints `CanonicalPath.decodeStoredMentionPath(chip.displayText)` over the chip's enclosing rect. Compact mode is gated on a single-line chip (`rects.count == 1` in `textChipDisplayMode`), so there's at most one rect per chip â€” don't call the label drawer across multiple line fragments.
