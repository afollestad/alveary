### Deep-Audit Follow-Ups (Resolved, 2026-04-06)

- [x] **`allow-unsigned-executable-memory` entitlement necessity** — validated via source analysis. Textual runs Prism.js through `JavaScriptCore` (`JSContext`), not `WKWebView`. Under Hardened Runtime without JIT entitlements, `JSContext` falls back to the Low-Level Interpreter (LLINT) which uses normal read-execute memory — no writable+executable mappings. The `allow-unsigned-executable-memory` entitlement is unrelated to JSC (that would be `allow-jit`), and `allow-jit` is also unnecessary because Prism.js tokenization is pure regex-based pattern matching that performs fine in the interpreter tier. The bundled `prism-bundle.js` is a static resource evaluated once at init — no `eval()`, no `new Function()`. **Conclusion**: keep the entitlement omitted. No plan changes needed.

- [x] **Claude graceful shutdown timing on `SIGTERM`** — validated across multiple scenarios (idle, mid-generation, mid-tool-use, interactive streaming mode). Claude exits **immediately** (≤1 second) on `SIGTERM` in all cases. Exit-code observations were **not** perfectly uniform: `-p` one-shot mode returned 143 (128 + SIGTERM signal 15), one active-turn long-lived `stream-json` probe also returned 143, and other interactive `--input-format stream-json` probes returned 0. The plan therefore treats fast exit as the invariant and does **not** key lifecycle handling off one exact exit code for long-lived sessions. The two-phase stop path (`SIGTERM` → grace period → `SIGKILL`) is safe; the `SIGKILL` fallback should rarely if ever fire. A 3–5 second grace period is more than sufficient. No plan changes needed.

- [x] **Textual `StructuredText` rendering performance with large markdown** — validated via source analysis of `gonzalezreal/textual`. **Key findings**: (1) `AttributedString(markdown:)` runs **synchronously on the main thread** with no caching — every content mutation re-parses the entire document; (2) `BlockVStack` is a custom `Layout` that measures **all blocks eagerly** (not lazy); (3) code block tokenization is async (renders plain text immediately, highlights later) but serialized through a single `CodeTokenizer` actor — multiple code blocks queue, not parallelize; (4) no cross-render cache for tokenization — streaming updates re-tokenize all changed blocks. **Impact for the app**: the outer `LazyVStack` helps at the message level, but each `StructuredText` view is internally eager. For completed messages (post-stream), a single 5,000-char render is acceptable. For streaming, the plan already accumulates the full assistant text and renders it as one `StructuredText` — this means every streamed token triggers a full re-parse. **Mitigation (already in plan)**: messages render as `StructuredText` only after streaming completes; during streaming, a lightweight `StreamingBubble` shows partial text without full markdown parsing. This two-phase approach avoids the re-parse-per-token cost. No plan changes needed.

- [x] **`git worktree add` with uncommitted changes in main repo** — validated with real git operations. `git worktree add -b new-branch <path> HEAD` succeeds (exit code 0) when the main repo has staged changes, unstaged modifications, and untracked files. Key findings: (1) uncommitted changes in the main repo are **not affected** — staged, unstaged, and untracked files remain exactly as they were after worktree creation; (2) the new worktree gets a **clean checkout** of the base ref — no staged/unstaged/untracked files from the main repo leak into the worktree; (3) worktree status is clean immediately after creation. This confirms `WorktreeManager.create()` is safe to call at any time regardless of the main repo's dirty state. No plan changes needed.

When a future provider needs a fresh process for the next turn, `AgentsManager` owns the re-spawn using the last successful `AgentSpawnConfig` and passes the turn via `initialPrompt`, so the replacement process is registered in the normal `processes`, `streamTasks`, and shutdown/status snapshots. Add focused unit tests when the first single-turn adapter is implemented.

- [x] **`AskUserQuestion` behavior in `-p --output-format stream-json` mode** — validated (CLI v2.1.92). The CLI **auto-denies** `AskUserQuestion` in `-p` mode, regardless of `--permission-mode`. Key findings: (1) The `tool_use` event IS emitted with the full structured input: `{"questions": [{"question": "...", "header": "...", "options": [{"label": "...", "description": "..."}], "multiSelect": false}]}`. (2) The CLI immediately returns a `tool_result` with `is_error: true` and `content: "Answer questions?"`. (3) The denial appears in the `result` event's `permission_denials` array with the full `tool_input` preserved. (4) The agent falls back to asking the question in plain text. **Architecture decision**: the app detects `AskUserQuestion` tool_calls in the event stream, renders a native SwiftUI selection UI (radio buttons / checkboxes from the structured options), and sends the user's selection as the next user message in a new turn. The agent's text fallback ("Waiting for your selection") naturally expects this follow-up. See Part 2b (Event Grouping) for the `ChatItem.promptBlock` type and Part 4d (Chat Blocks and Tool Rendering) for the selection UI rendering.
