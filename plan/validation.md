# Validation

Assumptions and architecture decisions that have been tested or need testing before implementation. No automated `validate-plan.sh` script is checked in here; re-run the targeted validations from the commands and notes captured below.

## Validated

- [x] **`--output-format stream-json` requires `--verbose`** — without `--verbose`, no output is produced. With it, structured JSON events stream correctly.
- [x] **Bidirectional streaming** — long-lived process with `--input-format stream-json` accepts multiple user messages via stdin JSON. No re-spawn needed between turns.
- [x] **Event JSON schemas** — `system/init`, `assistant` (text, tool_use, thinking), `user` (tool_result), and `result` events match the documented schemas. Verified with live CLI output.
- [x] **`--include-partial-messages`** — emits `stream_event` lines wrapping the Messages API streaming protocol (`message_start`, `content_block_delta`, `content_block_stop`, `message_stop`). Complete `assistant` event still arrives at the end.
- [x] **Session resume with `--session-id` + `--resume`** — create a session with `--session-id <uuid>`, then resume with `--resume <uuid>`. Agent retains full conversation context across sessions. Note: `--no-session-persistence` on the first turn prevents session file creation, making resume impossible.
- [x] **Mid-turn steering via stdin** — writing a JSON message to stdin while the agent is mid-tool-use is received and acted on. The agent finishes the in-flight tool call, reads the steering message, and redirects. Verified: sent "just say STEERED" during a Read tool call; agent complied in the second result.
- [x] **Permission modes** — `--permission-mode plan` allows read tools and simple bash commands but denies write tools (Claude declines in its response). `--permission-mode default` auto-denies write tools in non-interactive mode with an `is_error: true` tool result.
- [x] **`--model` flag** — accepts aliases (`opus`, `sonnet`, `haiku`) and full model IDs. The `system/init` event confirms the active model.
- [x] **Session path encoding** — sessions are stored at `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl` using Claude's canonicalized cwd encoding. Validation showed the directory is based on resolved `system/init.cwd` (`/private/...`, not a symlinked `/var/...`) and normalizes at least `/` and `.` to `-`, so the adapter should not assume slash replacement alone.
- [x] **SwiftData `@Query` with relationship predicate** — `#Predicate<Child> { $0.parent?.name == value }` compiles successfully. However, the app uses a denormalized `conversationId` field with a composite `#Index<ConversationEventRecord>([\.conversationId, \.timestamp])` instead for performance — relationship predicates require a join that degrades with thousands of records.
- [x] **Foundation `Process` + `terminationHandler` race** — setting `terminationHandler` after `run()` works even for fast-exiting processes. The handler fires correctly.
- [x] **Hook events** — `--include-hook-events` does not produce events in `-p` mode (tested CLI v2.1.92). Flag removed from spawn args. Agent state derived from standard events (`result`, `tool_result`, process lifecycle).
- [x] **Claude install URL** — `curl -fsSL https://claude.ai/install.sh | bash` is valid (302 redirect to GCS bootstrap script).

## Previously Unvalidated (Now Validated)

- [x] **`gh auth login --web --clipboard` non-interactive behavior** — validated end-to-end (logout → login → verify). Works without a TTY. Outputs one-time code and URL to stdout, copies code to clipboard, blocks until browser auth completes, exits 0 on success. Key finding: `--web` does NOT auto-open the browser without a TTY — the app must call `NSWorkspace.shared.open(url)` itself.
- [x] **`async let` with `readDataToEndOfFile()`** — validated: reading ~149KB stdout and stderr concurrently via `async let` works correctly. No deadlock, no data loss (all 2000 lines captured), process exits cleanly. Swift concurrency handles the blocking Foundation call without starving the thread pool.
- [x] **Markdown + syntax highlighting** — replaced MarkdownUI + Highlightr with Textual, which handles both markdown rendering and syntax highlighting (Prism.js, ~55 languages) in a single library.
- [x] **`--fork-session` for permission mode changes** — validated: `--resume <id> --fork-session --permission-mode <new>` preserves full conversation context (agent recalled a secret word) while switching permission mode and assigning a new session ID. The `system/init` event confirms the new mode and new session ID. The plan has been updated to use this approach instead of a simple kill/re-spawn.
- [x] **`--fork-session` for effort changes** — same mechanism as permission mode. `--effort` is a session-level spawn-time flag ("effort level for the current session" per `claude --help`), so changing it on a live session uses the same `--resume <id> --fork-session --effort <new>` flow. Context preservation is already validated above; the effort flag is simply an additional arg on re-spawn.
- [x] **Fork-session old/new branch semantics after session-ID rotation** — validated with real CLI probes. After `--resume <old> --fork-session ...`, Claude keeps **both** sessions resumable: the new session ID resumes the post-reconfigure branch, while the old session ID still resumes the pre-fork branch. This means a failed `updateSessionId()` persist after `system/init` is survivable for the current launch (the runtime can keep using the new ID in memory), but if the app relaunches before any later successful persist, the on-disk session map can silently reopen the stale pre-fork branch.
- [x] **Live post-fork Claude argv keeps the pre-fork resume ID** — validated on Claude Code v2.1.97 with a real `--resume <old> --fork-session` streaming process. After `system/init`, Claude reported a new `session_id` and all subsequent events used that new ID, but `ps -ww -p <pid> -o command=` still showed `--resume <old>` and did **not** contain the new ID. The plan now stores both the current resumable ID and the last launched argv ID in `SessionEntry`, so startup orphan cleanup can still prove ownership of a live post-reconfigure process.
- [x] **The first plain resume after a fork collapses argv/session identity back to the new branch ID** — validated on Claude Code v2.1.97 with a real `--resume <old> --fork-session` flow followed by a later plain `--resume <new>` streaming process. The forked child still advertised `--resume <old>` in `ps`, but the next ordinary resume reported `system/init.session_id = <new>`, `ps` showed `--resume <new>` (not the old ID), and the agent remembered state introduced only on the forked branch. The plan's `launchSessionId` split is therefore a temporary post-fork ownership bridge rather than a permanent dual-ID state.
- [x] **Later plain `persist()` repairs a failed `updateSessionId()` write** — validated with a throwaway Swift actor probe matching the documented `DefaultSessionManager` contract. After injecting a persist failure during `updateSessionId()`, the in-memory entry held the new forked session ID while disk still held the old one; a later successful plain `persist()` (no further session mutation required) wrote the repaired new ID to disk, and a fresh manager load preserved that repaired binding. Orderly shutdown's final `sessionManager.persist()` therefore acts as a real repair path when the app survives long enough to reach it.
- [x] **Shutdown repair-window headroom for oversized session maps** — validated with a throwaway Swift probe mirroring `DefaultSessionManager.persist()` (`JSONEncoder` + `data.write(..., .atomic)`) and the documented `applicationWillTerminate` bridge (`Task.detached` + `DispatchSemaphore.wait(timeout: .now() + 0.5)`). On local APFS, the repair path completed well within the 500ms budget for deliberately oversized maps: 50,000 entries (~20MB) in ~117ms, 100,000 entries (~39MB) in ~254ms, and 150,000 entries (~59MB) in ~377ms. The first timeout appeared around 200,000 entries (~79MB). This keeps the v1 500ms shutdown repair window comfortably above any plausible one-entry-per-conversation footprint, while still treating the path as best-effort rather than a hard guarantee on pathological filesystems.
- [x] **Resume after `SIGTERM` during an active turn** — validated on Claude Code v2.1.97 with a long-lived `--input-format stream-json` session. After starting a second turn, waiting for partial text, and then sending `SIGTERM`, Claude exited immediately and the same session ID resumed cleanly. In this specific probe the exit code was 143, but other interactive `stream-json` probes observed code 0, so the plan should not rely on one exact shutdown code for long-lived sessions. The resumed agent still remembered facts introduced by the interrupted user prompt, and the on-disk `.jsonl` had already appended that user message, but there was **no finalized assistant entry** for the interrupted turn before exit. Treat orderly shutdown/relaunch as preserving prior completed context plus committed user-turn state, not as guaranteeing that a partially streamed assistant response becomes a completed historical message.
- [x] **Queued stdin follow-ups are not durable across shutdown until Claude promotes them to an active turn** — validated on Claude Code v2.1.97 with a long-running first turn plus a second stdin JSON message injected while the first turn was still busy. Claude consumed the second write enough to append a `queue-operation` `enqueue` marker to its `.jsonl`, but it did **not** append a second `user` entry before `SIGTERM`, even when left queued for a couple more seconds while the first turn was still in-flight. After `--resume`, Claude remembered only the active first prompt. This sharpens the shutdown model: the active user turn is durable, queued-but-not-yet-started follow-ups are not, and partial assistant output is not finalized.
- [x] **Claude stale-session fallback after missing `.jsonl`** — validated with real session files. Deleting the actual `~/.claude/projects/<encoded-cwd>/<uuid>.jsonl` makes `--resume <uuid>` fail immediately with `No conversation found with session ID: ...`. Launching with `--session-id <same-uuid>` succeeds, recreates the `.jsonl`, and starts a fresh conversation with no stale history. Using `--session-id <same-uuid>` while the old file still exists fails with `Session ID ... is already in use.`, so the existence check is required.
- [x] **Live Claude process inspection for orphan cleanup** — validated on Claude Code v2.1.97. `ps -ww -p <pid> -o command=` exposes the full `--session-id <uuid>` / `--resume <uuid>` argv needed to recover the session binding from a live child, and `lsof -a -p <pid> -d cwd -Fn` reports the process cwd. When Claude is launched from a symlinked working directory, both the kernel-reported cwd and Claude's own `system/init.cwd` resolve to the canonical real path rather than the symlink alias. The plan now requires session-map `cwd` values and Claude stale-file checks to use that canonicalized path so startup orphan cleanup can match live processes against stored bindings.
- [x] **`tool_use_result` structured metadata** — validated (CLI v2.1.92). `user` / `tool_result` events can carry a sibling `tool_use_result` object with `stdout`, `stderr`, `interrupted`, `isImage`, and `noOutputExpected`. The plan now preserves these fields through `ConversationEvent.toolResult`, SwiftData, and tool rendering instead of dropping them at decode time.
- [x] **Custom `DiffParser` with real `git diff` output** — validated: the parser handles all tested edge cases (new files, deleted files, binary files, renames, permission-only changes, no-newline-at-EOF, multi-hunk, count-omitted hunk headers, empty diffs). A real 5,527-line git diff (3 files, 80 hunks) parsed correctly with accurate line numbers. The hunk header parser uses manual string parsing (`split`/`dropFirst`) instead of Swift regex literals — regex causes multi-minute compilation times.
- [x] **`NavigationSplitView` three-column layout on macOS 26** — validated: native three-column mode's `NavigationSplitViewVisibility` binding is non-functional for programmatic detail pane toggling. `HSplitView` works but produces incorrect animation direction. **Solution**: two-column `NavigationSplitView` (sidebar + detail) with `HStack` + conditional rendering + `withAnimation` + `.clipped()` inside the detail pane. This gives correct push-from-right animation for the diff viewer toggle.
- [x] **Knit code generation with XcodeGen project** — validated end-to-end. `KnitBuildPlugin` is Xcode-project-only and `knitconfig.json` is only for the plugin. The project uses `knit-cli gen` as a pre-build script phase instead. Key findings: (1) `knit-cli gen` requires `--assembly-input-path`, not `--config`; (2) the generated `KnitExtensions.swift` contains correct type-safe resolver extensions; (3) the full pipeline (XcodeGen → knit-cli gen → xcodebuild) compiles under Xcode 26.3 / Swift 6.2; (4) module-scope `ScopedModuleAssembler` requires `@MainActor` annotation to satisfy strict concurrency (non-`Sendable` type). Plan updated: pre-build script, code generation section, and `SkepApp` entry point.
- [x] **skills.sh API availability and format** — validated: API is live and stable. Response matches documented schema (`query`, `searchType`, `skills[]`, `count`, `duration_ms`). Skill entries have `id` (format: `owner/repo/skill-id`), `skillId`, `name`, `installs`, `source`. `searchType` is `"fuzzy"` (plan had `"semantic"` — corrected). Server enforces 2-char minimum (returns 400 for shorter queries). Empty results return `{"count": 0, "skills": []}`. Response time ~370ms.
- [x] **Sub-agent events stream individually in real-time** — validated: inner events stream as individual JSON lines during sub-agent execution (not batched). Also discovered three `system` event subtypes: `task_started` (sub-agent begins, includes `task_id`, `task_type`, `prompt`), `task_progress` (periodic, includes `description` (live status text), `last_tool_name`, token/tool count, duration), `task_notification` (completed, includes `status`, final usage). The `task_progress.description` field is a dynamic status message (e.g. "Searching for DiffParser") distinct from the Agent tool input description.
- [x] **`stream_event` ancestry for sub-agents** — validated: partial-message `stream_event` lines also carry `parent_tool_use_id`. The plan now keeps that ancestry on `.messageChunk` and only feeds top-level chunks into `streamingText`, preventing sub-agent streaming text from leaking into the main assistant bubble.

## Implementation Readiness

The current plan is sequentially buildable as written. Implementation should start at Phase 1 and preserve the already-resolved contracts around the Phase 1 `SkepApp`/`AppDelegate` bootstrap stub, Swift 6 actor isolation, `ModelContext` scoping, queue/send ordering, and lifecycle/shutdown ownership rather than re-opening those decisions in later phases. The remaining items below are implementation-time validation gates and future-provider preconditions, not blockers for starting Phase 1.
